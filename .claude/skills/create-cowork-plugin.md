# Cowork 플러그인 만들기

대화형 안내를 통해 처음부터 새 플러그인을 구축합니다. 발견 → 계획 → 설계 → 구현 → 패키징의 다섯 단계를 거쳐 설치 가능한 `.plugin` 파일을 최종 결과물로 제공합니다.

## 개요

플러그인은 커맨드, 스킬, 에이전트, 훅, MCP 서버 연동으로 Claude의 기능을 확장하는 독립적인 디렉토리입니다. 이 스킬은 전체 플러그인 아키텍처와 대화형으로 플러그인을 만들기 위한 5단계 워크플로우를 담고 있습니다.

진행 과정:
1. **발견(Discovery)** — 사용자가 무엇을 만들고 싶은지 파악
2. **컴포넌트 계획** — 필요한 컴포넌트 유형 결정
3. **설계 및 명확화 질문** — 각 컴포넌트의 세부 사항 지정
4. **구현** — 모든 플러그인 파일 생성
5. **검토 및 패키징** — `.plugin` 파일 전달

> **비기술적 출력**: 사용자와의 모든 대화는 평이한 언어로 진행합니다. 사용자가 요청하지 않는 한 파일 경로, 디렉토리 구조, 스키마 필드 등 구현 세부 사항은 노출하지 않습니다. 모든 내용은 플러그인이 무엇을 할 것인지 관점에서 설명합니다.

## 플러그인 아키텍처

### 디렉토리 구조

모든 플러그인은 다음 레이아웃을 따릅니다:

```
plugin-name/
├── .claude-plugin/
│   └── plugin.json           # 필수: 플러그인 매니페스트
├── commands/                 # 슬래시 커맨드 (.md 파일)
├── agents/                   # 서브에이전트 정의 (.md 파일)
├── skills/                   # 스킬 (SKILL.md가 있는 하위 디렉토리)
│   └── skill-name/
│       ├── SKILL.md
│       └── references/
├── .mcp.json                 # MCP 서버 정의
└── README.md                 # 플러그인 문서
```

**규칙:**
- `.claude-plugin/plugin.json`은 항상 필수입니다
- 컴포넌트 디렉토리(`commands/`, `agents/`, `skills/`)는 `.claude-plugin/` 안이 아닌 플러그인 루트에 위치합니다
- 플러그인이 실제로 사용하는 컴포넌트에 대한 디렉토리만 생성합니다
- 모든 디렉토리 및 파일명은 kebab-case를 사용합니다

### plugin.json 매니페스트

`.claude-plugin/plugin.json`에 위치합니다. 필수 필드는 `name`뿐입니다.

```json
{
  "name": "plugin-name",
  "version": "0.1.0",
  "description": "플러그인 목적에 대한 간단한 설명",
  "author": {
    "name": "작성자 이름"
  }
}
```

**이름 규칙:** kebab-case, 소문자와 하이픈, 공백이나 특수문자 없음.
**버전:** semver 형식 (MAJOR.MINOR.PATCH). `0.1.0`부터 시작합니다.

선택적 필드: `homepage`, `repository`, `license`, `keywords`.

커스텀 컴포넌트 경로를 지정할 수 있습니다 (자동 검색을 대체하지 않고 보완합니다):
```json
{
  "commands": "./custom-commands",
  "agents": ["./agents", "./specialized-agents"],
  "hooks": "./config/hooks.json",
  "mcpServers": "./.mcp.json"
}
```

### 컴포넌트 스키마

각 컴포넌트 유형에 대한 상세 스키마는 `references/component-schemas.md`에 있습니다. 요약:

| 컴포넌트 | 위치 | 형식 |
|---------|------|------|
| 커맨드 | `commands/*.md` | Markdown + YAML 프론트매터 |
| 스킬 | `skills/*/SKILL.md` | Markdown + YAML 프론트매터 |
| MCP 서버 | `.mcp.json` | JSON |
| 에이전트 (Cowork에서 드물게 사용) | `agents/*.md` | Markdown + YAML 프론트매터 |
| 훅 (Cowork에서 거의 사용 안 함) | `hooks/hooks.json` | JSON |

이 스키마는 Claude Code의 플러그인 시스템과 공유되지만, 지금 만드는 것은 지식 업무를 위한 데스크톱 앱인 Claude Cowork용 플러그인입니다.
Cowork 사용자에게는 커맨드와 스킬이 가장 유용합니다.

### `~~` 플레이스홀더를 활용한 커스터마이즈 가능한 플러그인

> **기본적으로 이 패턴을 사용하거나 질문하지 마세요.** 사용자가 조직 외부 사람들이 플러그인을 사용하기를 원한다고 명시적으로 말할 때만 `~~` 플레이스홀더를 도입합니다.
> 사용자가 외부에 플러그인을 배포하려는 것 같으면 이것이 옵션임을 언급할 수 있지만, AskUserQuestion으로 먼저 물어보지는 마세요.

플러그인을 회사 외부 사람들과 공유할 예정이라면, 개별 사용자에 맞게 조정이 필요한 부분이 있을 수 있습니다.
특정 제품 대신 카테고리로 외부 도구를 참조해야 할 수 있습니다 (예: "Jira" 대신 "프로젝트 트래커").
공유가 필요한 경우 일반적인 언어를 사용하고 `~~프로젝트 트래커에 이슈 생성`과 같이 물결표 두 개(`~~`)로 커스터마이즈가 필요한 부분을 표시합니다.
도구 카테고리를 사용한 경우, 플러그인 루트에 `CONNECTORS.md` 파일을 작성하여 설명합니다:

```markdown
# Connectors

## 도구 참조 방식

플러그인 파일은 `~~카테고리`를 해당 카테고리에서 사용자가 연결하는 도구의 플레이스홀더로 사용합니다.
플러그인은 도구에 종속되지 않으며, 특정 제품이 아닌 카테고리로 워크플로우를 설명합니다.

## 이 플러그인의 커넥터

| 카테고리 | 플레이스홀더 | 옵션 |
|---------|------------|------|
| 채팅 | `~~chat` | Slack, Microsoft Teams, Discord |
| 프로젝트 트래커 | `~~project tracker` | Linear, Asana, Jira |
```

### ${CLAUDE_PLUGIN_ROOT} 변수

훅과 MCP 설정에서 플러그인 내부 경로 참조 시 항상 `${CLAUDE_PLUGIN_ROOT}`를 사용합니다. 절대 경로를 하드코딩하지 마세요.

## 안내 워크플로우

사용자에게 질문할 때는 AskUserQuestion을 사용합니다. "업계 표준" 기본값이 맞다고 가정하지 마세요. 참고: AskUserQuestion에는 항상 건너뛰기 버튼과 자유 텍스트 입력 박스가 포함되므로 `없음` 또는 `기타`를 옵션으로 포함하지 마세요.

### 1단계: 발견(Discovery)

**목표**: 사용자가 무엇을 만들고 싶은지, 왜 만드는지 파악합니다.

물어볼 것 (불명확한 것만 — 사용자의 초기 요청에서 이미 답이 되었다면 건너뜁니다):

- 이 플러그인은 무엇을 해야 하나요? 어떤 문제를 해결하나요?
- 누가 어떤 상황에서 사용하나요?
- 외부 도구나 서비스와 연동이 필요한가요?
- 참고할 만한 유사한 플러그인이나 워크플로우가 있나요?

이해한 내용을 요약하고 진행 전 확인합니다.

**출력**: 플러그인의 목적과 범위에 대한 명확한 설명.

### 2단계: 컴포넌트 계획

**목표**: 플러그인에 필요한 컴포넌트 유형을 결정합니다.

발견 단계의 답변을 바탕으로 결정합니다:

- **스킬** — Claude가 필요에 따라 로드해야 하는 전문 지식이 있나요? (도메인 전문성, 참조 스키마, 워크플로우 가이드)
- **커맨드** — 사용자가 시작하는 액션이 있나요? (배포, 설정, 분석, 검토)
- **MCP 서버** — 외부 서비스 연동이 필요한가요? (데이터베이스, API, SaaS 도구)
- **에이전트 (드묾)** — 자율적인 다단계 작업이 있나요? (유효성 검사, 생성, 분석)
- **훅 (드묾)** — 특정 이벤트에서 자동으로 일어나야 할 일이 있나요? (정책 적용, 컨텍스트 로드, 작업 유효성 검사)

생성하지 않기로 결정한 컴포넌트 유형을 포함한 컴포넌트 계획 테이블을 제시합니다:

```
| 컴포넌트 | 수량 | 목적 |
|---------|------|------|
| 스킬    | 1    | X에 대한 도메인 지식 |
| 커맨드  | 2    | /do-thing, /check-thing |
| 에이전트| 0    | 불필요 |
| 훅      | 1    | 쓰기 유효성 검사 |
| MCP     | 1    | 서비스 Y 연결 |
```

진행 전 사용자 확인 또는 조정을 받습니다.

**출력**: 생성할 컴포넌트의 확정 목록.

### 3단계: 설계 및 명확화 질문

**목표**: 각 컴포넌트의 세부 사항을 지정합니다. 구현 전 모든 모호한 부분을 해소합니다.

계획에 있는 각 컴포넌트 유형에 대해 구체적인 설계 질문을 합니다. 컴포넌트 유형별로 질문을 그룹화하여 제시합니다. 진행 전 답변을 기다립니다.

**스킬:**
- 어떤 사용자 질의가 이 스킬을 트리거해야 하나요?
- 어떤 지식 도메인을 다루나요?
- 상세 내용을 위한 참조 파일을 포함해야 하나요?

**커맨드:**
- 각 커맨드는 어떤 인수를 받나요?
- 각 커맨드에 필요한 도구는 무엇인가요? (Read, Write, Bash, Grep 등)
- 각 커맨드는 대화형인가요, 자동화인가요?

**에이전트:**
- 각 에이전트는 능동적으로 트리거되어야 하나요, 요청 시에만 실행되어야 하나요?
- 어떤 도구가 필요한가요?
- 출력 형식은 어때야 하나요?

**훅:**
- 어떤 이벤트? (PreToolUse, PostToolUse, Stop, SessionStart 등)
- 어떤 동작 — 유효성 검사, 차단, 수정, 컨텍스트 추가?
- 프롬프트 기반(LLM 구동)인가요, 커맨드 기반(결정론적 스크립트)인가요?

**MCP 서버:**
- 서버 유형은? (로컬용 stdio, OAuth가 있는 호스팅 서버용 SSE, REST API용 HTTP)
- 인증 방법은?
- 어떤 도구가 노출되어야 하나요?

사용자가 "알아서 결정해주세요"라고 하면 구체적인 권장사항을 제시하고 명시적 확인을 받습니다.

**출력**: 모든 컴포넌트에 대한 상세 사양.

### 4단계: 구현

**목표**: 모범 사례에 따라 모든 플러그인 파일을 생성합니다.

**작업 순서:**
1. 플러그인 디렉토리 구조 생성
2. `plugin.json` 매니페스트 생성
3. 각 컴포넌트 생성 (정확한 형식은 `references/component-schemas.md` 참조)
4. 플러그인을 문서화하는 `README.md` 생성

**구현 지침:**

- **커맨드**는 사용자에게 보내는 메시지가 아니라 Claude에게 내리는 지시입니다. 무엇을 할지 지시문 형태로 작성합니다.
- **스킬**은 점진적 공개 방식을 사용합니다: SKILL.md 본문은 간결하게 (3,000 단어 이하), 상세 내용은 `references/`에. 프론트매터 description은 3인칭으로 특정 트리거 문구를 포함해야 합니다.
- **에이전트**는 트리거 조건을 보여주는 `<example>` 블록이 있는 description과 마크다운 본문의 시스템 프롬프트가 필요합니다.
- **훅** 설정은 `hooks/hooks.json`에 위치합니다. 스크립트 경로에는 `${CLAUDE_PLUGIN_ROOT}`를 사용합니다. 복잡한 로직에는 프롬프트 기반 훅을 권장합니다.
- **MCP 설정**은 플러그인 루트의 `.mcp.json`에 위치합니다. 로컬 서버 경로에는 `${CLAUDE_PLUGIN_ROOT}`를 사용합니다. README에 필요한 환경 변수를 문서화합니다.

### 5단계: 검토 및 패키징

**목표**: 완성된 플러그인을 전달합니다.

1. 생성된 내용 요약 — 각 컴포넌트와 목적을 나열합니다
2. 사용자가 조정을 원하는지 묻습니다
3. `claude plugin validate <path-to-plugin-json>` 실행; 오류와 경고를 수정합니다
4. `.plugin` 파일로 패키징:

```bash
cd /path/to/plugin-dir && zip -r /tmp/plugin-name.plugin . -x "*.DS_Store" && cp /tmp/plugin-name.plugin /path/to/outputs/plugin-name.plugin
```

> **중요**: 항상 먼저 `/tmp/`에 zip을 생성한 다음 outputs 폴더에 복사합니다. outputs 폴더에 직접 쓰면 권한 문제로 실패할 수 있습니다.

> **이름 지정**: `.plugin` 파일에는 `plugin.json`의 플러그인 이름을 사용합니다 (예: 이름이 `code-reviewer`이면 `code-reviewer.plugin`으로 출력).

`.plugin` 파일은 채팅에 풍부한 미리보기로 나타나며, 사용자는 파일을 탐색하고 버튼을 눌러 플러그인을 수락할 수 있습니다.

## 모범 사례

- **작게 시작**: 최소한의 컴포넌트 세트로 시작합니다. 잘 만들어진 스킬 하나가 엉성한 컴포넌트 다섯 개보다 유용합니다.
- **스킬의 점진적 공개**: SKILL.md에 핵심 지식을, `references/`에 상세 참조 자료를, `examples/`에 실제 예시를.
- **명확한 트리거 문구**: 스킬 description에는 사용자가 말할 구체적인 문구를 포함합니다. 에이전트 description에는 `<example>` 블록을 포함합니다.
- **커맨드는 Claude를 위한 것**: 커맨드 내용은 사용자가 읽을 문서가 아닌 Claude가 따를 지시문으로 작성합니다.
- **명령형 문체**: 스킬에서 동사 우선 지시문을 사용합니다 ("설정 파일을 파싱합니다", "설정 파일을 파싱해야 합니다" 아님).
- **이식성**: 플러그인 내부 경로에는 항상 `${CLAUDE_PLUGIN_ROOT}`를 사용하고 하드코딩된 경로는 절대 사용하지 않습니다.
- **보안**: 자격 증명에는 환경 변수를 사용하고, 원격 서버에는 HTTPS를 사용하며, 최소 권한 도구 접근을 적용합니다.

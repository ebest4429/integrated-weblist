# 데이터 탐색 스킬

데이터셋 프로파일링, 품질 평가, 패턴 발견, 스키마 이해를 위한 체계적인 방법론입니다.

## 데이터 프로파일링 방법론

### Phase 1: 구조 파악

데이터를 분석하기 전에 구조를 먼저 이해합니다:

**테이블 수준 질문:**
- 행과 열의 수는?
- 단위(grain)는 무엇인가? (1행 = 무엇?)
- 기본 키는 무엇인가? 고유한가?
- 데이터는 언제 마지막으로 갱신되었는가?
- 데이터가 얼마나 오래 전까지 있는가?

**컬럼 분류:**
각 컬럼을 다음 중 하나로 분류합니다:
- **식별자(Identifier)**: 고유 키, 외래 키, 엔티티 ID
- **차원(Dimension)**: 그룹화/필터링용 범주형 속성 (상태, 유형, 지역, 카테고리)
- **지표(Metric)**: 측정용 수치 값 (매출, 건수, 기간, 점수)
- **시간(Temporal)**: 날짜 및 타임스탬프 (created_at, updated_at, event_date)
- **텍스트(Text)**: 자유형 텍스트 필드 (설명, 메모, 이름)
- **불리언(Boolean)**: 참/거짓 플래그
- **구조체(Structural)**: JSON, 배열, 중첩 구조

### Phase 2: 컬럼 수준 프로파일링

각 컬럼에 대해 다음을 계산합니다:

**모든 컬럼:**
- NULL 개수 및 NULL 비율
- 고유값 수 및 카디널리티 비율 (고유값 수 / 전체)
- 가장 빈번한 값 (상위 5~10개, 빈도 포함)
- 가장 드문 값 (하위 5개, 이상 징후 탐지용)

**숫자형 컬럼 (지표):**
```
최솟값, 최댓값, 평균, 중앙값(p50)
표준편차
백분위수: p1, p5, p25, p75, p95, p99
0 개수
음수 개수 (예상치 못한 경우)
```

**문자열 컬럼 (차원, 텍스트):**
```
최소 길이, 최대 길이, 평균 길이
빈 문자열 개수
패턴 분석 (값이 특정 형식을 따르는가?)
대소문자 일관성 (전체 대문자, 전체 소문자, 혼합?)
앞뒤 공백 개수
```

**날짜/타임스탬프 컬럼:**
```
최소 날짜, 최대 날짜
NULL 날짜
미래 날짜 (예상치 못한 경우)
월/주 단위 분포
시계열 갭
```

**불리언 컬럼:**
```
true 개수, false 개수, null 개수
true 비율
```

### Phase 3: 관계 탐색

개별 컬럼 프로파일링 후:

- **외래 키 후보**: 다른 테이블과 연결될 수 있는 ID 컬럼
- **계층 구조**: 자연스러운 드릴다운 경로를 형성하는 컬럼 (국가 > 시도 > 시군구)
- **상관관계**: 함께 움직이는 수치형 컬럼
- **파생 컬럼**: 다른 컬럼으로부터 계산된 것으로 보이는 컬럼
- **중복 컬럼**: 동일하거나 거의 동일한 정보를 가진 컬럼

## 품질 평가 프레임워크

### 완전성 점수

각 컬럼을 다음과 같이 평가합니다:
- **완전** (non-null 99% 초과): 녹색
- **대부분 완전** (95~99%): 노란색 — NULL 값 조사 필요
- **불완전** (80~95%): 주황색 — 이유와 영향 파악 필요
- **희소** (80% 미만): 빨간색 — 대체(imputation) 없이는 사용 불가

### 일관성 확인

다음 항목을 점검합니다:
- **값 형식 불일치**: 동일한 개념의 다양한 표현 ("대한민국", "한국", "Korea", "KR")
- **유형 불일치**: 문자열로 저장된 숫자, 다양한 형식의 날짜
- **참조 무결성**: 부모 레코드가 없는 외래 키
- **비즈니스 규칙 위반**: 음수 수량, 시작일보다 이른 종료일, 100% 초과 퍼센트
- **컬럼 간 일관성**: status = "completed"인데 completed_at이 NULL

### 정확성 지표

정확성 문제를 시사하는 위험 신호:
- **플레이스홀더 값**: 0, -1, 999999, "N/A", "TBD", "test", "xxx"
- **기본값**: 특정 값의 비정상적으로 높은 빈도
- **오래된 데이터**: 활성 시스템인데 updated_at에 최근 변경 없음
- **불가능한 값**: 150세 초과 나이, 먼 미래 날짜, 음수 기간
- **반올림 편향**: 모든 값이 0 또는 5로 끝남 (측정값이 아닌 추정값 시사)

### 적시성 평가

- 테이블은 언제 마지막으로 갱신되었는가?
- 예상 갱신 주기는?
- 이벤트 발생 시간과 로드 시간 사이에 지연이 있는가?
- 시계열에 갭이 있는가?

## 패턴 발견 기법

### 분포 분석

수치형 컬럼의 분포 특성:
- **정규분포**: 평균과 중앙값이 가깝고 종 모양
- **우측 편향**: 높은 값의 긴 꼬리 (매출, 세션 기간에서 흔함)
- **좌측 편향**: 낮은 값의 긴 꼬리 (덜 흔함)
- **이봉분포**: 두 개의 피크 (두 개의 뚜렷한 모집단 시사)
- **멱 법칙**: 극히 큰 값 소수 + 작은 값 다수 (사용자 활동에서 흔함)
- **균등분포**: 구간 전체에 걸쳐 거의 동일한 빈도 (합성 또는 무작위 데이터에서 흔함)

### 시간적 패턴

시계열 데이터에서 탐색할 사항:
- **추세**: 지속적인 상승 또는 하락
- **계절성**: 반복 패턴 (주간, 월간, 분기, 연간)
- **요일 효과**: 평일 vs. 주말 차이
- **공휴일 효과**: 알려진 공휴일 전후 감소 또는 급등
- **변화점**: 수준 또는 추세의 급격한 변화
- **이상값**: 패턴을 벗어나는 개별 데이터 포인트

### 세그먼트 탐색

자연스러운 세그먼트 파악 방법:
- 고유값이 3~20개인 범주형 컬럼 탐색
- 세그먼트 값별 지표 분포 비교
- 행동 패턴이 현저히 다른 세그먼트 탐색
- 세그먼트 내 동질성 또는 하위 세그먼트 존재 여부 확인

### 상관관계 탐색

수치형 컬럼 간:
- 모든 지표 쌍에 대한 상관 행렬 계산
- 강한 상관관계 (|r| > 0.7) 플래그 후 조사
- 주의: 상관관계는 인과관계를 의미하지 않음 — 명시적으로 표시할 것
- 비선형 관계 확인 (예: 이차, 로그)

## 스키마 이해 및 문서화

### 스키마 문서화 템플릿

팀 공유용 데이터셋 문서화 시:

```markdown
## 테이블: [schema.table_name]

**설명**: [이 테이블이 나타내는 것]
**단위(Grain)**: [1행 = ...]
**기본 키**: [컬럼]
**행 수**: [대략적인 수, 기준 날짜 포함]
**갱신 주기**: [실시간 / 시간별 / 일별 / 주별]
**담당자**: [담당 팀 또는 담당자]

### 주요 컬럼

| 컬럼 | 유형 | 설명 | 예시 값 | 비고 |
|------|------|------|---------|------|
| user_id | STRING | 고유 사용자 식별자 | "usr_abc123" | users.id의 FK |
| event_type | STRING | 이벤트 유형 | "click", "view", "purchase" | 고유값 15개 |
| revenue | DECIMAL | USD 기준 거래 매출 | 29.99, 149.00 | 비구매 이벤트는 NULL |
| created_at | TIMESTAMP | 이벤트 발생 시간 | 2024-01-15 14:23:01 | 이 컬럼 기준 파티션 |

### 관계
- `users`와 `user_id`로 조인
- `products`와 `product_id`로 조인
- `event_details`의 부모 (event_id 기준 1:다)

### 알려진 이슈
- [알려진 데이터 품질 이슈 목록]
- [분석가를 위한 주의사항]

### 일반 쿼리 패턴
- [이 테이블의 일반적인 사용 사례]
```

### 스키마 탐색 쿼리

데이터 웨어하우스에 연결된 경우 스키마 탐색에 사용하는 패턴:

```sql
-- 스키마 내 모든 테이블 목록 (PostgreSQL)
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

-- 컬럼 상세 정보 (PostgreSQL)
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'my_table'
ORDER BY ordinal_position;

-- 테이블 크기 (PostgreSQL)
SELECT relname, pg_size_pretty(pg_total_relation_size(relid))
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC;

-- 모든 테이블 행 수 (일반 패턴)
-- 테이블별 실행: SELECT COUNT(*) FROM table_name
```

### 계보(Lineage)와 의존성

생소한 데이터 환경 탐색 시:

1. "출력" 테이블(보고서나 대시보드에서 사용하는 테이블)부터 시작
2. 업스트림 추적: 어떤 테이블이 이것을 공급하는가?
3. raw / staging / mart 레이어 파악
4. 원시 데이터에서 분석 테이블까지의 변환 체인 매핑
5. 데이터가 보강, 필터링, 집계되는 지점 기록

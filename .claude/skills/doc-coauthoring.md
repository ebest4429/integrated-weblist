# 문서 공동 작성 워크플로우

이 스킬은 협업 문서 작성을 단계별로 안내하는 구조화된 워크플로우를 제공합니다. 적극적인 가이드 역할을 수행하며, 세 단계를 통해 사용자를 이끕니다: 맥락 수집, 구성 및 다듬기, 독자 테스트.

## 이 워크플로우를 제안하는 경우

**실행 조건:**
- 사용자가 문서 작성을 언급하는 경우: "문서 작성해줘", "제안서 초안 만들어줘", "명세서 만들어줘", "정리해줘"
- 사용자가 특정 문서 유형을 언급하는 경우: "PRD", "설계 문서", "의사결정 문서", "RFC"
- 사용자가 상당한 분량의 글쓰기 작업을 시작하려는 것으로 보이는 경우

**초기 제안:**
사용자에게 문서 공동 작성을 위한 구조화된 워크플로우를 제안합니다. 세 단계를 설명합니다:

1. **맥락 수집**: 사용자가 관련 맥락을 제공하는 동안 Claude가 명확화 질문을 합니다
2. **구성 및 다듬기**: 브레인스토밍과 편집을 통해 각 섹션을 반복적으로 구성합니다
3. **독자 테스트**: 맥락이 없는 새 Claude(컨텍스트 없음)로 문서를 테스트하여 다른 사람이 읽기 전에 놓친 부분을 발견합니다

이 방식이 다른 사람이 문서를 읽을 때(Claude에 붙여넣을 때 포함) 잘 작동하도록 돕는다고 설명합니다. 이 워크플로우를 사용할지, 자유형식으로 진행할지 물어봅니다.

사용자가 거절하면 자유형식으로 작업합니다. 수락하면 1단계로 진행합니다.

## 1단계: 맥락 수집

**목표:** 사용자가 알고 있는 것과 Claude가 아는 것 사이의 간극을 좁혀, 이후 단계에서 더 나은 안내를 제공할 수 있도록 합니다.

### 초기 질문

문서에 대한 메타 맥락을 질문합니다:

1. 이 문서의 유형은 무엇인가요? (예: 기술 명세서, 의사결정 문서, 제안서)
2. 주요 독자는 누구인가요?
3. 이 문서를 읽은 후 독자에게 기대하는 반응이나 행동은 무엇인가요?
4. 따라야 할 템플릿이나 특정 형식이 있나요?
5. 알아두어야 할 제약 사항이나 기타 맥락이 있나요?

단축어를 사용하거나 편한 방식으로 자유롭게 답해도 된다고 안내합니다.

**사용자가 템플릿을 제공하거나 문서 유형을 언급하는 경우:**
- 공유할 템플릿 문서가 있는지 질문합니다
- 공유 문서 링크를 제공하면 해당 통합 기능을 사용하여 가져옵니다
- 파일을 제공하면 읽습니다

**사용자가 기존 공유 문서를 편집하는 경우:**
- 해당 통합 기능을 사용하여 현재 상태를 읽습니다
- 대체 텍스트(alt-text)가 없는 이미지를 확인합니다
- 대체 텍스트가 없는 이미지가 있으면, 다른 사람이 Claude를 사용하여 문서를 이해할 때 해당 이미지를 볼 수 없다고 설명합니다. 대체 텍스트 생성 여부를 물어봅니다. 원하는 경우, 각 이미지를 채팅에 붙여넣어 설명적인 대체 텍스트를 생성할 수 있도록 요청합니다.

### 정보 덤프

초기 질문에 답한 후, 가지고 있는 모든 맥락을 자유롭게 덤프하도록 권장합니다. 다음과 같은 정보를 요청합니다:
- 프로젝트/문제에 대한 배경
- 팀 관련 논의 또는 공유 문서
- 대안적 솔루션을 채택하지 않은 이유
- 조직적 맥락 (팀 역학, 과거 사건, 조직 정치)
- 일정 압박이나 제약 사항
- 기술 아키텍처 또는 의존성
- 이해관계자 우려 사항

정리하지 않아도 되니 그냥 모두 털어놓으라고 안내합니다. 맥락 제공 방법을 여러 가지 제시합니다:
- 생각 흐름대로 자유롭게 서술
- 팀 채널이나 스레드 링크 제공
- 공유 문서 링크 제공

**통합 기능이 사용 가능한 경우** (예: Slack, Teams, Google Drive, SharePoint 또는 기타 MCP 서버), 이를 활용하여 맥락을 직접 가져올 수 있다고 언급합니다.

**통합 기능이 감지되지 않고 Claude.ai 또는 Claude 앱을 사용하는 경우:** Claude 설정에서 커넥터를 활성화하면 메시징 앱과 문서 저장소에서 직접 맥락을 가져올 수 있다고 제안합니다.

초기 덤프가 완료되면 명확화 질문을 할 것이라고 안내합니다.

**맥락 수집 중:**

- 사용자가 팀 채널이나 공유 문서를 언급하는 경우:
  - 통합 기능이 사용 가능한 경우: 지금 내용을 읽겠다고 알린 후 해당 통합 기능 사용
  - 통합 기능이 사용 불가한 경우: 접근 권한 없음을 설명. Claude 설정에서 커넥터 활성화를 제안하거나, 관련 내용을 직접 붙여넣도록 안내

- 사용자가 알 수 없는 주체/프로젝트를 언급하는 경우:
  - 연결된 도구에서 검색하면 더 알아볼 수 있는지 질문
  - 사용자 확인을 받은 후에만 검색 진행

- 사용자가 맥락을 제공하는 동안, 파악된 내용과 아직 불분명한 내용을 추적

**명확화 질문:**

사용자가 초기 덤프를 완료했다는 신호를 보내면(또는 상당한 맥락이 제공된 후), 이해도를 높이기 위한 명확화 질문을 합니다:

맥락의 빈틈을 바탕으로 5~10개의 번호가 매겨진 질문을 생성합니다.

단축어로 답해도 된다고 안내합니다 (예: "1: 맞아, 2: #채널 참고, 3: 하위 호환성 때문에 안 됨"). 더 많은 문서 링크나 채널 링크를 제공하거나 계속 정보를 덤프해도 됩니다. 가장 효율적인 방법을 선택하도록 안내합니다.

**종료 조건:**
기본적인 설명 없이도 엣지 케이스와 트레이드오프에 대한 질문이 가능할 때 충분한 맥락이 수집된 것입니다.

**전환:**
더 추가할 맥락이 있는지, 아니면 문서 초안 작성으로 넘어갈 준비가 됐는지 질문합니다.

더 추가하고 싶으면 계속하도록 합니다. 준비가 되면 2단계로 진행합니다.

## 2단계: 구성 및 다듬기

**목표:** 브레인스토밍, 선별, 반복적 개선을 통해 섹션별로 문서를 구성합니다.

**사용자에게 설명:**
문서를 섹션별로 구성해 나간다고 설명합니다. 각 섹션에 대해:
1. 포함할 내용에 대한 명확화 질문을 합니다
2. 5~20개의 옵션을 브레인스토밍합니다
3. 사용자가 유지/제거/통합할 항목을 선택합니다
4. 섹션 초안을 작성합니다
5. 세밀한 편집을 통해 다듬습니다

가장 미지수가 많은 섹션(보통 핵심 결정/제안)부터 시작하고, 나머지를 순서대로 진행합니다.

**섹션 순서:**

문서 구조가 명확한 경우:
어떤 섹션부터 시작할지 질문합니다.

미지수가 가장 많은 섹션부터 시작하도록 제안합니다. 의사결정 문서의 경우 보통 핵심 제안이 해당되고, 명세서의 경우 기술적 접근 방식이 해당됩니다. 요약 섹션은 마지막에 작성하는 것이 좋습니다.

사용자가 어떤 섹션이 필요한지 모르는 경우:
문서 유형과 템플릿을 바탕으로 3~5개의 적합한 섹션을 제안합니다.

이 구조가 맞는지, 조정이 필요한지 질문합니다.

**구조가 합의되면:**

모든 섹션에 대한 placeholder 텍스트가 포함된 초기 문서 구조를 생성합니다.

**아티팩트에 접근 가능한 경우:**
`create_file`을 사용하여 아티팩트를 생성합니다. 이를 통해 Claude와 사용자 모두 작업할 기반 구조를 갖습니다.

모든 섹션의 placeholder가 포함된 초기 구조를 생성할 것이라고 안내합니다.

모든 섹션 헤더와 "[작성 예정]" 또는 "[내용 입력]" 같은 간단한 placeholder 텍스트가 포함된 아티팩트를 생성합니다.

scaffold 링크를 제공하고 각 섹션 채우기를 시작할 때라고 알립니다.

**아티팩트에 접근 불가한 경우:**
작업 디렉토리에 마크다운 파일을 생성합니다. 적절한 이름으로 명명합니다 (예: `decision-doc.md`, `technical-spec.md`).

모든 섹션의 placeholder가 포함된 초기 구조를 생성할 것이라고 안내합니다.

모든 섹션 헤더와 placeholder 텍스트가 포함된 파일을 생성합니다.

파일이 생성되었음을 확인하고 각 섹션 채우기를 시작할 때라고 알립니다.

**각 섹션에 대해:**

### 1단계: 명확화 질문

[섹션 이름] 섹션 작업을 시작한다고 알립니다. 포함할 내용에 대한 5~10개의 명확화 질문을 합니다:

맥락과 섹션 목적을 바탕으로 5~10개의 구체적인 질문을 생성합니다.

단축어로 답하거나 중요한 것을 언급하기만 해도 된다고 안내합니다.

### 2단계: 브레인스토밍

[섹션 이름] 섹션에 포함될 수 있는 것들을 섹션의 복잡도에 따라 5~20개 브레인스토밍합니다. 다음을 찾아봅니다:
- 공유된 맥락 중 잊혀진 것들
- 아직 언급되지 않은 관점이나 고려 사항

섹션 복잡도에 따라 5~20개의 번호가 매겨진 옵션을 생성합니다. 마지막에 더 많은 옵션이 필요하면 추가로 브레인스토밍할 수 있다고 제안합니다.

### 3단계: 선별

유지, 제거, 또는 통합할 항목을 선택해 달라고 요청합니다. 다음 섹션에서 우선순위를 파악하는 데 도움이 되므로 간략한 이유도 함께 요청합니다.

예시 제공:
- "1,4,7,9 유지"
- "3 제거 (1과 중복)"
- "6 제거 (독자가 이미 알고 있는 내용)"
- "11과 12 통합"

**사용자가 번호 선택 대신 자유형식으로 피드백하는 경우** (예: "좋아요" 또는 "대부분 괜찮은데..."), 선호도를 파악하여 진행합니다. 유지/제거/변경할 내용을 파악하고 적용합니다.

### 4단계: 빈틈 확인

선택한 내용을 바탕으로 [섹션 이름] 섹션에서 중요하게 빠진 내용이 있는지 질문합니다.

### 5단계: 초안 작성

`str_replace`를 사용하여 이 섹션의 placeholder 텍스트를 실제 초안 내용으로 교체합니다.

선택한 내용을 바탕으로 [섹션 이름] 섹션 초안을 작성하겠다고 알립니다.

**아티팩트를 사용하는 경우:**
초안 작성 후 아티팩트 링크를 제공합니다.

읽어보고 변경할 내용을 알려달라고 요청합니다. 구체적으로 알려줄수록 다음 섹션 학습에 도움이 된다고 안내합니다.

**파일을 사용하는 경우 (아티팩트 없음):**
초안 작성 완료를 확인합니다.

[파일명]에 [섹션 이름] 섹션 초안이 작성되었음을 알립니다. 읽어보고 변경할 내용을 알려달라고 요청합니다. 구체적으로 알려줄수록 다음 섹션 학습에 도움이 된다고 안내합니다.

**첫 번째 섹션 초안 작성 시 사용자에게 안내할 핵심 지침:**
문서를 직접 편집하는 대신, 변경할 내용을 알려달라고 요청합니다. 이를 통해 스타일을 파악하여 다음 섹션에 반영할 수 있습니다. 예: "X 항목 제거 - Y에서 이미 다룸" 또는 "세 번째 단락을 더 간결하게 해줘"

### 6단계: 반복적 개선

사용자 피드백에 따라:
- `str_replace`를 사용하여 편집 (전체 문서 재출력 금지)
- **아티팩트를 사용하는 경우:** 각 편집 후 아티팩트 링크 제공
- **파일을 사용하는 경우:** 편집 완료만 확인
- 사용자가 문서를 직접 편집하고 읽어달라고 하면: 변경 사항을 파악하고 향후 섹션에서 참고 (선호도 반영)

사용자가 섹션에 만족할 때까지 **계속 반복합니다**.

### 품질 확인

3번 연속으로 실질적인 변경 없이 반복되면, 중요한 정보를 잃지 않고 제거할 수 있는 내용이 있는지 질문합니다.

섹션이 완료되면 [섹션 이름] 완료를 확인하고 다음 섹션으로 넘어갈 준비가 됐는지 질문합니다.

**모든 섹션에 대해 반복합니다.**

### 완성 단계

전체 섹션의 80% 이상이 완료되면, 전체 문서를 다시 읽고 다음을 확인하겠다고 알립니다:
- 섹션 간 흐름과 일관성
- 중복이나 모순
- 일반적인 내용 채우기식 "슬롭" 느낌
- 모든 문장이 의미를 담고 있는지

전체 문서를 읽고 피드백을 제공합니다.

**모든 섹션 초안 작성 및 개선 완료 후:**
모든 섹션 초안이 완성되었음을 알립니다. 완성된 문서를 한 번 더 검토하겠다고 알립니다.

전체적인 일관성, 흐름, 완성도를 검토합니다.

최종 제안을 제공합니다.

독자 테스트로 넘어갈 준비가 됐는지, 아니면 더 다듬고 싶은 내용이 있는지 질문합니다.

## 3단계: 독자 테스트

**목표:** 맥락이 없는 새 Claude(컨텍스트 없음)로 문서를 테스트하여 독자에게 실제로 작동하는지 검증합니다.

**사용자에게 설명:**
문서가 독자에게 실제로 작동하는지 테스트할 것이라고 설명합니다. 이를 통해 작성자에게는 명확하지만 다른 사람에게는 혼란스러울 수 있는 부분을 발견합니다.

### 테스트 방법

**서브 에이전트에 접근 가능한 경우 (예: Claude Code):**

사용자 개입 없이 직접 테스트를 수행합니다.

### 1단계: 독자 질문 예측

독자가 이 문서를 발견하려 할 때 어떤 질문을 할지 예측하겠다고 알립니다.

독자가 실제로 물어볼 만한 5~10개의 질문을 생성합니다.

### 2단계: 서브 에이전트로 테스트

이 대화의 맥락 없이 새 Claude 인스턴스로 이 질문들을 테스트하겠다고 알립니다.

각 질문에 대해 문서 내용과 질문만으로 서브 에이전트를 호출합니다.

각 질문에 대해 독자 Claude가 맞게/틀리게 답한 내용을 요약합니다.

### 3단계: 추가 검사

추가 검사를 수행하겠다고 알립니다.

서브 에이전트를 호출하여 모호함, 잘못된 가정, 모순을 확인합니다.

발견된 문제를 요약합니다.

### 4단계: 보고 및 수정

문제가 발견된 경우:
독자 Claude가 특정 부분에서 어려움을 겪었다고 보고합니다.

구체적인 문제 목록을 나열합니다.

해당 빈틈을 수정하겠다고 알립니다.

문제가 있는 섹션에 대해 다듬기 단계로 돌아갑니다.

---

**서브 에이전트에 접근 불가한 경우 (예: claude.ai 웹 인터페이스):**

사용자가 수동으로 테스트를 진행해야 합니다.

### 1단계: 독자 질문 예측

이 문서를 발견하려 할 때 어떤 질문을 할지 물어봅니다. Claude.ai에 무엇을 입력할 것 같은지 질문합니다.

독자가 실제로 물어볼 만한 5~10개의 질문을 생성합니다.

### 2단계: 테스트 설정

테스트 지침을 제공합니다:
1. 새 Claude 대화 열기: https://claude.ai
2. 문서 내용 붙여넣기 또는 공유 (커넥터가 활성화된 공유 문서 플랫폼을 사용하는 경우 링크 제공)
3. 독자 Claude에게 생성된 질문 질문하기

각 질문에 대해 독자 Claude에게 다음을 제공하도록 지시합니다:
- 답변
- 모호하거나 불명확한 부분이 있는지
- 문서가 독자가 이미 알고 있다고 가정하는 지식이나 맥락

독자 Claude가 정확하게 답하는지, 아니면 잘못 해석하는지 확인합니다.

### 3단계: 추가 확인

독자 Claude에게 다음도 질문합니다:
- "이 문서에서 독자에게 모호하거나 불명확할 수 있는 부분은 무엇인가요?"
- "이 문서가 독자가 이미 알고 있다고 가정하는 지식이나 맥락은 무엇인가요?"
- "내부 모순이나 불일치가 있나요?"

### 4단계: 결과 기반 반복

독자 Claude가 틀리거나 어려워했던 부분을 알려달라고 요청합니다. 해당 빈틈을 수정하겠다고 알립니다.

문제가 있는 섹션에 대해 다듬기 단계로 돌아갑니다.

---

### 종료 조건 (두 방식 모두 해당)

독자 Claude가 질문에 일관되게 정확히 답하고 새로운 빈틈이나 모호함이 발견되지 않으면 문서가 준비된 것입니다.

## 최종 검토

독자 테스트 통과 시:
문서가 독자 Claude 테스트를 통과했음을 알립니다. 완료 전:

1. 스스로 최종 검토를 권장합니다 — 이 문서의 오너는 사용자이며 품질에 대한 책임도 사용자에게 있습니다
2. 사실, 링크, 기술적 세부 사항을 다시 확인하도록 제안합니다
3. 원했던 효과를 달성하는지 검증하도록 요청합니다

한 번 더 검토를 원하는지, 아니면 작업이 완료되었는지 질문합니다.

**최종 검토를 원하면 수행합니다. 아니라면:**
문서 완성을 알립니다. 몇 가지 최종 팁을 제공합니다:
- 부록에 이 대화 링크를 추가하면 독자들이 문서 작성 과정을 확인할 수 있습니다
- 부록을 활용하여 본문을 늘리지 않고 깊이를 더할 수 있습니다
- 실제 독자로부터 피드백을 받으면 문서를 업데이트합니다

## 효과적인 안내를 위한 팁

**톤:**
- 직접적이고 절차적으로 안내합니다
- 사용자 행동에 영향을 미칠 때만 이유를 간략히 설명합니다
- 접근 방식을 "판매"하려 하지 말고 그냥 실행합니다

**이탈 처리:**
- 사용자가 단계를 건너뛰려는 경우: 이 단계를 건너뛰고 자유형식으로 작성하겠냐고 질문합니다
- 사용자가 답답해 보이는 경우: 예상보다 오래 걸리고 있음을 인정합니다. 더 빠르게 진행하는 방법을 제안합니다
- 항상 사용자에게 프로세스를 조정할 권한을 줍니다

**맥락 관리:**
- 언급된 내용에 대한 맥락이 부족하면 적극적으로 질문합니다
- 빈틈이 누적되지 않도록 발견되는 즉시 해결합니다

**아티팩트 관리:**
- 전체 섹션 초안 작성에는 `create_file`을 사용합니다
- 모든 편집에는 `str_replace`를 사용합니다
- 모든 변경 후 아티팩트 링크를 제공합니다
- 브레인스토밍 목록에는 아티팩트를 사용하지 않습니다 — 그냥 대화로 진행합니다

**속도보다 품질:**
- 단계를 서둘러 진행하지 않습니다
- 각 반복마다 의미 있는 개선이 이루어져야 합니다
- 목표는 독자에게 실제로 작동하는 문서를 만드는 것입니다

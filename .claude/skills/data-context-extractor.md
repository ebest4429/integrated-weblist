# Data Context Extractor

분석가로부터 회사 고유의 데이터 지식을 추출하고 맞춤형 데이터 분석 스킬을 생성하는 메타 스킬입니다.

## 작동 방식

이 스킬은 두 가지 모드로 동작합니다:

1. **부트스트랩 모드**: 새로운 데이터 분석 스킬을 처음부터 생성
2. **반복 모드**: 도메인별 참조 파일을 추가하여 기존 스킬 개선

---

## 부트스트랩 모드

사용 시점: 사용자가 웨어하우스용 새 데이터 컨텍스트 스킬을 생성하려는 경우.

### Phase 1: 데이터베이스 연결 및 탐색

**1단계: 데이터베이스 유형 파악**

질문: "어떤 데이터 웨어하우스를 사용하고 계신가요?"

주요 옵션:
- **BigQuery**
- **Snowflake**
- **PostgreSQL/Redshift**
- **Databricks**

`~~data warehouse` 도구(쿼리 및 스키마)를 사용하여 연결합니다. 불명확한 경우 현재 세션에서 사용 가능한 MCP 도구를 확인합니다.

**2단계: 스키마 탐색**

`~~data warehouse` 스키마 도구를 사용하여:
1. 사용 가능한 데이터셋/스키마 목록 조회
2. 가장 중요한 테이블 파악 (사용자에게 질문: "분석가가 가장 자주 쿼리하는 테이블 3~5개는 무엇인가요?")
3. 해당 핵심 테이블의 스키마 상세 정보 수집

방언별 탐색 쿼리 예시:
```sql
-- BigQuery: 데이터셋 목록
SELECT schema_name FROM INFORMATION_SCHEMA.SCHEMATA

-- BigQuery: 데이터셋 내 테이블 목록
SELECT table_name FROM `project.dataset.INFORMATION_SCHEMA.TABLES`

-- Snowflake: 스키마 목록
SHOW SCHEMAS IN DATABASE my_database

-- Snowflake: 테이블 목록
SHOW TABLES IN SCHEMA my_schema
```

### Phase 2: 핵심 질문 (반드시 수행)

스키마 탐색 후 다음 질문들을 대화 방식으로 진행합니다 (한꺼번에 묻지 않음):

**엔티티 명확화 (중요)**
> "여기서 '사용자' 또는 '고객'이라고 할 때 정확히 무엇을 의미하나요? 유형이 여러 가지인가요?"

주의 깊게 들을 사항:
- 복수의 엔티티 유형 (user vs account vs organization)
- 상호 관계 (1:1, 1:다, 다:다)
- 연결에 사용되는 ID 필드

**기본 식별자**
> "[고객/사용자/계정]의 주요 식별자는 무엇인가요? 동일 엔티티에 ID가 여러 개인 경우가 있나요?"

주의 깊게 들을 사항:
- 기본 키 vs 비즈니스 키
- UUID vs 정수형 ID
- 레거시 ID 체계

**핵심 지표**
> "가장 자주 묻는 지표 2~3가지는 무엇인가요? 각각 어떻게 계산되나요?"

주의 깊게 들을 사항:
- 정확한 공식 (ARR = monthly_revenue × 12)
- 각 지표에 연관된 테이블/컬럼
- 기간 관례 (최근 7일, 달력 월 등)

**데이터 위생**
> "쿼리에서 항상 필터링해야 하는 데이터는 무엇인가요? (테스트 데이터, 사기, 내부 사용자 등)"

주의 깊게 들을 사항:
- 항상 포함해야 할 표준 WHERE 절
- 제외 여부를 나타내는 플래그 컬럼 (is_test, is_internal, is_fraud)
- 제외할 특정 값 (status = 'deleted')

**흔한 함정**
> "신입 분석가들이 이 데이터로 주로 어떤 실수를 하나요?"

주의 깊게 들을 사항:
- 헷갈리는 컬럼명
- 타임존 문제
- NULL 처리 특이사항
- 과거 이력 테이블 vs 현재 상태 테이블

### Phase 3: 스킬 생성

다음 구조로 스킬을 생성합니다:

```
[company]-data-analyst/
├── SKILL.md
└── references/
    ├── entities.md          # 엔티티 정의 및 관계
    ├── metrics.md           # KPI 계산식
    ├── tables/              # 도메인별 파일
    │   ├── [domain1].md
    │   └── [domain2].md
    └── dashboards.json      # 선택사항: 기존 대시보드 카탈로그
```

**SKILL.md 템플릿**: `references/skill-template.md` 참조

**SQL 방언 섹션**: `references/sql-dialects.md` 참조하여 적절한 방언 내용 포함

**참조 파일 템플릿**: `references/domain-template.md` 참조

### Phase 4: 패키징 및 전달

1. 스킬 디렉토리에 모든 파일 생성
2. zip 파일로 패키징
3. 수집된 내용 요약과 함께 사용자에게 전달

## 반복 모드

사용 시점: 사용자가 기존 스킬을 보유하고 있지만 추가 컨텍스트가 필요한 경우.

### 1단계: 기존 스킬 로드

사용자에게 기존 스킬(zip 또는 폴더)을 업로드하도록 요청하거나, 세션에 이미 있는 경우 찾습니다.

현재 SKILL.md와 참조 파일을 읽어 이미 문서화된 내용을 파악합니다.

### 2단계: 갭 파악

질문: "어떤 도메인이나 주제에 더 많은 컨텍스트가 필요한가요? 어떤 쿼리가 실패하거나 잘못된 결과를 내고 있나요?"

일반적인 갭:
- 새로운 데이터 도메인 (마케팅, 재무, 제품 등)
- 누락된 지표 정의
- 문서화되지 않은 테이블 관계
- 새로운 용어

### 3단계: 타겟 탐색

파악된 도메인에 대해:

1. **관련 테이블 탐색**: `~~data warehouse` 스키마 도구로 해당 도메인의 테이블 검색
2. **도메인별 질문**:
   - "[도메인] 분석에 어떤 테이블을 사용하나요?"
   - "[도메인]의 핵심 지표는 무엇인가요?"
   - "[도메인] 데이터에 특별한 필터나 주의사항이 있나요?"

3. **새 참조 파일 생성**: 도메인 템플릿을 사용하여 `references/[domain].md` 생성

### 4단계: 업데이트 및 재패키징

1. 새 참조 파일 추가
2. SKILL.md의 "지식 베이스 탐색" 섹션에 새 도메인 포함
3. 스킬 재패키징
4. 업데이트된 스킬을 사용자에게 전달

## 참조 파일 기준

각 참조 파일에 포함해야 할 사항:

### 테이블 문서화
- **위치**: 전체 테이블 경로
- **설명**: 이 테이블에 포함된 내용, 사용 시점
- **기본 키**: 행을 고유하게 식별하는 방법
- **갱신 주기**: 데이터 갱신 빈도
- **주요 컬럼**: 컬럼명, 유형, 설명, 비고가 포함된 표
- **관계**: 다른 테이블과의 조인 방법
- **샘플 쿼리**: 자주 사용되는 쿼리 패턴 2~3개

### 지표 문서화
- **지표명**: 사람이 읽기 쉬운 이름
- **정의**: 평이한 언어로 설명
- **공식**: 컬럼 참조가 포함된 정확한 계산식
- **소스 테이블**: 데이터 출처
- **주의사항**: 엣지 케이스, 제외 항목, 함정

### 엔티티 문서화
- **엔티티명**: 불리는 이름
- **정의**: 비즈니스 내에서 나타내는 것
- **기본 테이블**: 이 엔티티를 찾는 위치
- **ID 필드**: 식별 방법
- **관계**: 다른 엔티티와의 관련성
- **일반 필터**: 표준 제외 항목 (내부, 테스트 등)

## 품질 체크리스트

생성된 스킬 전달 전 확인:

- [ ] SKILL.md에 완전한 프론트매터 포함 (name, description)
- [ ] 엔티티 명확화 섹션이 명확함
- [ ] 핵심 용어가 정의되어 있음
- [ ] 표준 필터/제외 항목이 문서화됨
- [ ] 도메인별 샘플 쿼리 최소 2~3개
- [ ] SQL이 올바른 방언 구문 사용
- [ ] 참조 파일이 SKILL.md 탐색 섹션에 링크됨

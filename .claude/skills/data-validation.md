# 데이터 검증 스킬

배포 전 QA 체크리스트, 데이터 분석 흔한 함정, 결과 합리성 검사, 재현성을 위한 문서화 기준을 제공합니다.

## 배포 전 QA 체크리스트

이해관계자에게 분석 결과를 공유하기 전에 이 체크리스트를 실행합니다.

### 데이터 품질 확인

- [ ] **소스 검증**: 사용된 테이블/데이터 소스 확인. 이 질문에 적합한 소스인가?
- [ ] **최신성**: 분석에 충분히 최신 데이터. "기준 날짜" 명시.
- [ ] **완전성**: 시계열에 예상치 못한 갭이나 누락된 세그먼트 없음.
- [ ] **NULL 처리**: 주요 컬럼의 NULL 비율 확인. NULL이 적절히 처리됨 (제외, 대체, 또는 표시).
- [ ] **중복 제거**: 잘못된 조인 또는 중복 소스 레코드로 인한 이중 카운팅 없음.
- [ ] **필터 검증**: 모든 WHERE 절과 필터가 올바름. 의도치 않은 제외 없음.

### 계산 확인

- [ ] **집계 로직**: GROUP BY에 집계되지 않은 모든 컬럼 포함. 집계 수준이 분석 단위와 일치.
- [ ] **분모 정확성**: 비율 및 퍼센트 계산에 올바른 분모 사용. 분모가 0이 아님.
- [ ] **날짜 정렬**: 비교 시 동일한 기간 길이 사용. 불완전한 기간은 제외하거나 명시.
- [ ] **조인 정확성**: JOIN 유형이 적절함 (INNER vs LEFT). 다대다 조인으로 건수가 증가하지 않음.
- [ ] **지표 정의**: 지표가 이해관계자의 정의와 일치. 차이가 있으면 명시.
- [ ] **소계 합산**: 부분의 합이 전체와 일치 (기대되는 경우). 일치하지 않으면 이유 설명 (예: 중복).

### 합리성 확인

- [ ] **크기**: 숫자가 합리적인 범위 내. 매출이 음수가 아님. 퍼센트가 0~100% 사이.
- [ ] **추세 연속성**: 시계열에 설명되지 않은 급등락 없음.
- [ ] **교차 검증**: 주요 수치가 다른 알려진 소스와 일치 (대시보드, 이전 보고서, 재무 데이터).
- [ ] **규모 수준**: 총 매출이 적절한 수준. 사용자 수가 알려진 수치와 일치.
- [ ] **엣지 케이스**: 경계에서는 어떻게 되는가? 빈 세그먼트, 활동 없는 기간, 신규 엔티티.

### 발표 확인

- [ ] **차트 정확성**: 막대 차트는 0에서 시작. 축에 레이블 있음. 패널 간 스케일 일관성.
- [ ] **숫자 형식**: 적절한 정밀도. 일관된 통화/퍼센트 형식. 필요한 경우 천 단위 구분자.
- [ ] **제목 명확성**: 제목이 단순 지표가 아닌 인사이트를 나타냄. 날짜 범위 명시.
- [ ] **주의사항 투명성**: 알려진 한계와 가정을 명시적으로 기술.
- [ ] **재현성**: 제공된 문서만으로 다른 사람이 이 분석을 재현할 수 있음.

## 데이터 분석 흔한 함정

### 조인 폭발 (Join Explosion)

**문제**: 다대다 조인이 조용히 행을 증가시켜 건수와 합계를 부풀림.

**탐지 방법**:
```sql
-- 조인 전후 행 수 확인
SELECT COUNT(*) FROM table_a;  -- 1,000
SELECT COUNT(*) FROM table_a a JOIN table_b b ON a.id = b.a_id;  -- 3,500 (문제 발생)
```

**방지 방법**:
- 조인 후 항상 행 수 확인
- 건수가 증가하면 조인 관계 조사 (실제로 1:1인가, 1:다인가?)
- 조인을 통해 엔티티를 카운팅할 때 `COUNT(*)` 대신 `COUNT(DISTINCT a.id)` 사용

### 생존 편향 (Survivorship Bias)

**문제**: 현재 존재하는 엔티티만 분석하고, 삭제되거나 이탈하거나 실패한 것들을 무시.

**사례**:
- "현재 사용자"의 행동 분석은 이탈한 사용자를 놓침
- "우리 제품을 사용 중인 회사"를 살펴보면 평가 후 떠난 회사를 무시
- "성공한" 결과의 속성을 "실패한" 결과 없이 연구

**방지 방법**: 결론을 내리기 전에 "이 데이터셋에 포함되지 않은 사람은 누구인가?"를 질문.

### 불완전 기간 비교 (Incomplete Period Comparison)

**문제**: 불완전한 기간과 완전한 기간을 비교.

**사례**:
- "1월 매출은 50만 달러 vs. 12월 80만 달러" — 그런데 1월이 아직 끝나지 않음
- "이번 주 가입자가 감소" — 수요일에 확인, 이전 완전한 한 주와 비교

**방지 방법**: 항상 완전한 기간으로 필터링하거나, 동일 일(月중 동일 날짜) 또는 동일 일수로 비교.

### 분모 변동 (Denominator Shifting)

**문제**: 기간 간 분모가 변경되어 비율 비교가 불가능해짐.

**사례**:
- "적격 사용자" 카운트 방식을 변경하여 전환율이 개선된 것처럼 보임
- "활성" 정의가 업데이트되어 이탈률이 변경됨

**방지 방법**: 비교하는 모든 기간에 일관된 정의 사용. 정의 변경 사항 명시.

### 평균의 평균 (Average of Averages)

**문제**: 그룹 크기가 다를 때 미리 계산된 평균을 다시 평균 내면 잘못된 결과.

**사례**:
- 그룹 A: 사용자 100명, 평균 매출 50달러
- 그룹 B: 사용자 10명, 평균 매출 200달러
- 잘못된 계산: 평균의 평균 = (50 + 200) / 2 = 125달러
- 올바른 계산: 가중 평균 = (100×50 + 10×200) / 110 = 63.64달러

**방지 방법**: 항상 원시 데이터에서 집계. 사전 집계된 평균을 다시 평균 내지 말 것.

### 타임존 불일치 (Timezone Mismatches)

**문제**: 서로 다른 데이터 소스가 다른 타임존을 사용하여 정렬 오류 발생.

**사례**:
- UTC 기준 이벤트 타임스탬프 vs. 현지 시간 기준 사용자 노출 날짜
- 서로 다른 마감 시간을 사용하는 일별 롤업

**방지 방법**: 분석 전에 모든 타임스탬프를 단일 타임존(UTC 권장)으로 표준화. 사용된 타임존 문서화.

### 세그먼테이션 선택 편향 (Selection Bias in Segmentation)

**문제**: 측정하려는 결과로 세그먼트를 정의하면 순환 논리 발생.

**사례**:
- "온보딩을 완료한 사용자는 리텐션이 더 높다" — 당연히, 그들이 스스로 선택한 것
- "파워 유저는 더 많은 매출을 창출한다" — 매출을 창출했기 때문에 파워 유저가 됨

**방지 방법**: 결과가 아닌 처치 전(pre-treatment) 특성을 기반으로 세그먼트 정의.

## 결과 합리성 검사

### 크기 확인

분석의 모든 주요 수치에 대해 "냄새 테스트(smell test)" 통과 여부 확인:

| 지표 유형 | 합리성 확인 |
|---------|-----------|
| 사용자 수 | 알려진 MAU/DAU 수치와 일치하는가? |
| 매출 | 알려진 ARR 대비 적절한 규모인가? |
| 전환율 | 0%~100% 사이인가? 대시보드 수치와 일치하는가? |
| 성장률 | MoM 50% 이상 성장이 현실적인가, 아니면 데이터 문제인가? |
| 평균 | 분포에 대한 알고 있는 내용을 고려할 때 평균이 합리적인가? |
| 퍼센트 | 세그먼트 퍼센트의 합이 ~100%인가? |

### 교차 검증 기법

1. **동일한 지표를 두 가지 다른 방법으로 계산**하여 일치 여부 확인
2. **개별 레코드 점검** — 특정 엔티티 몇 개를 선택하여 데이터를 수동으로 추적
3. **알려진 벤치마크와 비교** — 공개된 대시보드, 재무 보고서, 이전 분석과 대조
4. **역산** — 총 매출이 X라면, 1인당 매출 × 사용자 수가 X에 근접하는가?
5. **경계 확인** — 단일 날짜, 단일 사용자, 단일 카테고리로 필터링하면 어떻게 되는가? 그 미시적 결과가 합리적인가?

### 조사가 필요한 위험 신호

- 명확한 원인 없이 전기 대비 50% 이상 변화한 지표
- 정확히 반올림된 숫자인 건수 또는 합계 (필터 또는 기본값 문제 시사)
- 정확히 0% 또는 100%인 비율 (불완전한 데이터 가능성)
- 가설을 완벽하게 확인하는 결과 (현실은 보통 더 복잡함)
- 기간 또는 세그먼트에 걸쳐 동일한 값 (쿼리가 특정 차원을 무시하고 있을 가능성)

## 재현성을 위한 문서화 기준

### 분석 문서화 템플릿

모든 비자명한 분석에 포함해야 할 사항:

```markdown
## 분석: [제목]

### 질문
[답변하려는 구체적인 질문]

### 데이터 소스
- 테이블: [schema.table_name] (기준일: [날짜])
- 테이블: [schema.other_table] (기준일: [날짜])
- 파일: [파일명] (소스: [출처])

### 정의
- [지표 A]: [계산 방법 상세]
- [세그먼트 X]: [소속 조건 상세]
- [기간]: [시작일] ~ [종료일], [타임존]

### 방법론
1. [분석 접근 방법 1단계]
2. [2단계]
3. [3단계]

### 가정 및 한계
- [가정 1과 합리적인 이유]
- [한계 1과 결론에 대한 잠재적 영향]

### 핵심 발견
1. [근거가 있는 발견 1]
2. [근거가 있는 발견 2]

### SQL 쿼리
[사용된 모든 쿼리, 주석 포함]

### 주의사항
- [독자가 이 내용에 따라 행동하기 전에 알아야 할 사항]
```

### 코드 문서화

재사용될 수 있는 코드(SQL, Python)의 경우:

```python
"""
분석: 월별 코호트 리텐션
작성자: [이름]
날짜: [날짜]
데이터 소스: events 테이블, users 테이블
마지막 검증: [날짜] — 결과가 대시보드와 2% 이내로 일치

목적:
    첫 활동 날짜 기준 월별 사용자 리텐션 코호트 계산.

가정:
    - "활성"은 해당 월에 최소 1개의 이벤트를 의미
    - 테스트/내부 계정 제외 (user_type != 'internal')
    - 전체 UTC 날짜 사용

출력:
    cohort_month 행과 months_since_signup 열로 구성된 코호트 리텐션 매트릭스.
    값은 리텐션 비율 (0~100%).
"""
```

### 분석의 버전 관리

- 쿼리 및 코드를 버전 관리(git) 또는 공유 문서 시스템에 저장
- 사용된 데이터 스냅샷 날짜 기록
- 갱신된 데이터로 분석을 재실행하는 경우 변경된 내용과 이유 문서화
- 추세 비교를 위해 반복 분석의 이전 버전에 링크

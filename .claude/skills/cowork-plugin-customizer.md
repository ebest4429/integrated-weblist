# Cowork 플러그인 커스터마이징

특정 조직을 위해 플러그인을 커스터마이징합니다 — 일반 플러그인 템플릿을 처음 설정하거나, 이미 설정된 플러그인을 수정하고 개선하는 작업 모두 지원합니다.

> **플러그인 찾기**: 플러그인 소스 파일을 찾으려면 `find mnt/.local-plugins mnt/.plugins -type d -name "*<plugin-name>*"` 명령으로 플러그인 디렉토리를 찾은 후, 파일을 읽어 구조를 파악한 뒤 변경 작업을 시작합니다. 플러그인 디렉토리를 찾을 수 없는 경우, 사용자가 원격 컨테이너에서 실행 중일 가능성이 있습니다. 작업을 중단하고 다음과 같이 안내하세요: "플러그인 커스터마이징은 현재 데스크톱 앱의 Cowork 모드에서만 사용할 수 있습니다."

## 커스터마이징 모드 결정

플러그인을 찾은 후 `~~` 접두사 플레이스홀더 여부를 확인합니다: `grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'`

> **기본 규칙**: `~~` 플레이스홀더가 존재하면, 사용자가 특정 부분만 커스터마이징하도록 명시적으로 요청하지 않는 한 **일반 플러그인 설정** 모드로 기본 진행합니다.

**1. 일반 플러그인 설정** — 플러그인에 `~~` 접두사 플레이스홀더가 있는 경우. 이는 템플릿의 커스터마이징 지점으로, 실제 값으로 교체해야 합니다 (예: `~~Jira` → `Asana`, `~~your-team-channel` → `#engineering`).

**2. 범위 한정 커스터마이징** — `~~` 플레이스홀더가 없고, 사용자가 특정 부분의 커스터마이징을 요청한 경우 (예: "커넥터 커스터마이징", "스탠드업 명령 업데이트", "티켓 도구 변경"). 관련 섹션만 읽고 해당 부분에만 집중합니다. 플러그인 전체를 스캔하거나 관련 없는 항목을 제시하지 않습니다.

**3. 전체 커스터마이징** — `~~` 플레이스홀더가 없고, 사용자가 플러그인을 전반적으로 수정하려는 경우. 플러그인 파일을 읽어 현재 설정을 파악한 후, 변경하고 싶은 내용을 사용자에게 묻습니다.

> **중요**: 커스터마이징 중인 플러그인 또는 스킬의 이름을 절대 변경하지 마세요. 디렉토리, 파일, 플러그인/스킬 이름 필드를 변경하지 않습니다.

> **비기술적 출력**: 사용자에게 표시되는 모든 출력(할 일 목록, 질문, 요약)은 평이하고 비기술적인 언어로 작성해야 합니다. `~~` 접두사, 플레이스홀더, 커스터마이징 포인트에 대해 사용자에게 언급하지 마세요. 모든 내용을 플러그인의 기능과 조직의 도구 관점에서 설명합니다.

## 커스터마이징 워크플로우

### Phase 0: 사용자 의도 파악 (범위 한정 및 전체 커스터마이징에만 해당)

**범위 한정 커스터마이징**과 **전체 커스터마이징** (일반 플러그인 설정 제외)의 경우, 사용자가 요청과 함께 자유 형식의 컨텍스트를 제공했는지 확인합니다 (예: "스탠드업 명령 커스터마이징 — 우리는 매일 아침 #eng-updates에서 비동기 스탠드업을 합니다").

- **사용자가 컨텍스트를 제공한 경우**: 기록하고 Phase 3에서 답변을 미리 채우는 데 활용합니다 — 사용자가 이미 답한 질문은 다시 묻지 않습니다.
- **사용자가 컨텍스트를 제공하지 않은 경우**: 진행 전에 AskUserQuestion으로 개방형 질문 하나를 합니다. 요청한 커스터마이징 내용에 맞게 질문을 조정하세요 (예: "brief 명령에 어떤 변경을 원하시나요?" 또는 "이 플러그인의 동작 방식에서 무엇을 바꾸고 싶으신가요?"). 짧고 요청에 구체적으로 작성합니다.

  응답(있는 경우)을 이후 단계 전반에 걸쳐 추가 컨텍스트로 활용합니다.

### Phase 1: 지식 MCP에서 컨텍스트 수집

회사 내부 지식 MCP를 사용하여 커스터마이징 범위와 관련된 정보를 수집합니다. 카테고리별 상세 쿼리 패턴은 `references/search-strategies.md`를 참조하세요.

**수집 대상** (관련 있는 항목만):
- 조직이 사용하는 도구 이름 및 서비스
- 조직의 프로세스 및 워크플로우
- 팀 관례 (명명 규칙, 상태값, 추정 척도)
- 설정 값 (워크스페이스 ID, 프로젝트 이름, 팀 식별자)

**검색 소스:**
1. **채팅/Slack MCP** — 도구 언급, 통합, 워크플로우 논의
2. **문서 MCP** — 온보딩 문서, 도구 가이드, 설정 지침
3. **이메일 MCP** — 라이선스 알림, 관리자 이메일, 설정 초대

모든 결과를 기록하여 Phase 3에서 활용합니다.

### Phase 2: 할 일 목록 작성

변경해야 할 항목의 할 일 목록을 적절한 범위로 구성합니다:

- **범위 한정 커스터마이징**: 사용자가 요청한 특정 섹션과 관련된 항목만 포함합니다.
- **일반 플러그인 설정**: `grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'`으로 모든 플레이스홀더 커스터마이징 포인트를 찾습니다. 테마별로 그룹화합니다.
- **전체 커스터마이징**: 플러그인 파일을 읽어 현재 설정을 파악하고, 사용자 요청을 바탕으로 변경이 필요한 항목을 식별합니다.

플러그인의 목적에 초점을 맞춘 사용자 친화적 설명을 사용합니다:

- **좋은 예**: "회사의 스탠드업 준비 방식 파악"
- **나쁜 예**: "commands/standup-prep.md의 플레이스홀더 교체"

### Phase 3: 할 일 항목 처리

Phase 0과 Phase 1의 컨텍스트를 활용하여 각 항목을 처리합니다.

**사용자의 자유 형식 입력(Phase 0) 또는 지식 MCP(Phase 1)에서 명확한 답을 얻은 경우**: 확인 없이 바로 적용합니다.

**그렇지 않은 경우**: AskUserQuestion을 사용합니다. "업계 표준" 기본값이 맞다고 가정하지 마세요 — 사용자 입력이나 지식 MCP에서 구체적인 답을 얻지 못했다면 묻습니다. 참고: AskUserQuestion은 항상 건너뛰기 버튼과 자유 입력창을 포함하므로 `없음` 또는 `기타` 옵션을 선택지에 추가하지 않습니다.

**변경 유형:**

1. **플레이스홀더 교체** (일반 설정): `~~Jira` → `Asana`, `~~your-org-channel` → `#engineering`
2. **콘텐츠 업데이트**: 조직에 맞게 지침, 명령, 워크플로우, 참조 수정
3. **URL 패턴 업데이트**: `tickets.example.com/your-team/123` → `app.asana.com/0/PROJECT_ID/TASK_ID`
4. **설정 값**: 워크스페이스 ID, 프로젝트 이름, 팀 식별자

사용자가 모르거나 건너뛰는 경우, 값을 변경하지 않고 유지합니다 (일반 설정의 경우 `~~` 접두사 플레이스홀더 그대로 유지).

### Phase 4: 유용한 MCP 검색

커스터마이징 항목이 처리된 후, 식별되거나 변경된 도구에 대한 MCP를 연결합니다. 전체 워크플로우, 카테고리별 키워드 매핑, 설정 파일 형식은 `references/mcp-servers.md`를 참조하세요.

커스터마이징 중 식별된 각 도구에 대해:
1. 레지스트리 검색: `search_mcp_registry(keywords=[...])` — `references/mcp-servers.md`의 카테고리 키워드 사용, 또는 이미 알고 있는 경우 특정 도구명으로 검색
2. 연결되지 않은 경우: `suggest_connectors(directoryUuids=["chosen-uuid"])` — 사용자가 인증 완료
3. 플러그인의 MCP 설정 파일 업데이트 (`plugin.json`에서 사용자 지정 위치 확인, 없으면 루트의 `.mcp.json`)

모든 MCP 결과를 수집하여 요약 출력에 함께 제시합니다 (아래 참조) — 이 단계에서 MCP를 하나씩 제시하지 않습니다.

## 플러그인 패키징

모든 커스터마이징이 적용된 후, 플러그인을 `.plugin` 파일로 패키징합니다:

1. **플러그인 디렉토리 압축** (`setup/`은 더 이상 필요하지 않으므로 제외):
   ```bash
   cd /path/to/plugin && zip -r /tmp/plugin-name.plugin . -x "setup/*" && cp /tmp/plugin-name.plugin /path/to/outputs/plugin-name.plugin
   ```
2. **`.plugin` 확장자로 파일을 사용자에게 제시**하여 직접 설치할 수 있게 합니다. (.plugin 파일을 제시하면 사용자에게 플러그인 파일을 둘러볼 수 있는 리치 미리보기로 표시되며, 버튼을 눌러 커스터마이징을 수락할 수 있습니다.)

> **중요**: 항상 `/tmp/`에 먼저 zip을 생성한 후 outputs 폴더로 복사합니다. outputs 폴더에 직접 쓰면 권한 문제로 실패하고 임시 파일이 남을 수 있습니다.

> **이름 지정**: 원래 플러그인 디렉토리 이름을 `.plugin` 파일에 사용합니다 (예: 플러그인 디렉토리가 `coder`이면 출력 파일은 `coder.plugin`). 커스터마이징 중 플러그인이나 파일 이름을 변경하지 마세요 — 플레이스홀더 값 교체와 콘텐츠 업데이트만 수행합니다.

## 요약 출력

커스터마이징 후, 소스별로 그룹화된 학습 내용 요약을 사용자에게 제시합니다. 설정 중 연결된 MCP와 아직 연결해야 할 MCP를 보여주는 섹션을 항상 포함합니다:

```markdown
## Slack 검색 결과
- 프로젝트 관리 도구로 Asana를 사용합니다
- 스프린트 주기는 2주입니다

## 문서 검색 결과
- 스토리 포인트는 티셔츠 사이즈를 사용합니다

## 직접 답변 내용
- 티켓 상태: Backlog, In Progress, In Review, Done
```

그 다음 설정 중 연결된 MCP와 아직 연결해야 할 MCP를 연결 방법과 함께 제시합니다.

Phase 1에서 지식 MCP를 사용할 수 없었고 사용자가 하나 이상의 질문에 수동으로 답해야 했다면, 마지막에 다음 안내를 추가합니다:
> 참고로, Slack 또는 Microsoft Teams와 같은 소스를 연결하면 다음에 플러그인을 커스터마이징할 때 자동으로 답을 찾을 수 있습니다.

# Skill Creator (스킬 제작자)

새로운 스킬을 만들고 반복적으로 개선하기 위한 스킬입니다.

전체 프로세스를 개략적으로 설명하면 다음과 같습니다:

- 스킬이 무엇을 해야 하는지, 어떻게 해야 하는지 대략적으로 결정
- 스킬 초안 작성
- 몇 가지 테스트 프롬프트를 만들고 "스킬을 가진 claude"로 실행
- 결과를 정성적·정량적으로 평가하도록 사용자 지원
  - 실행이 백그라운드에서 진행되는 동안, 아직 없다면 정량적 평가(eval)를 초안으로 작성 (있다면 그대로 사용하거나 필요시 수정). 사용자에게 설명
  - `eval-viewer/generate_review.py` 스크립트로 결과를 보여주고, 정량 지표도 확인
- 평가 결과에 따라 스킬 수정
- 만족할 때까지 반복
- 테스트 세트를 확장하고 더 큰 규모로 다시 시도

이 스킬을 사용할 때 여러분의 역할은 사용자가 이 과정의 어느 단계에 있는지 파악하고 그 단계에서 진행을 도와주는 것입니다. 예를 들어 "X를 위한 스킬을 만들고 싶어요"라고 하면, 의도를 구체화하고, 초안을 쓰고, 테스트 케이스를 작성하고, 평가 방법을 정하고, 모든 프롬프트를 실행하고, 반복합니다.

이미 스킬 초안이 있는 경우에는 바로 평가/반복 단계로 넘어가면 됩니다.

물론 "많은 평가 실행은 필요 없고 감각적으로 해봐요"라고 하면 그렇게 해도 됩니다. 유연하게 대처하세요.

스킬이 완성된 후(순서는 유연하게), 스킬 설명 최적화 도구도 실행할 수 있습니다. 이 도구는 별도 스크립트로 스킬 트리거 정확도를 최적화합니다.

## 사용자와의 소통

Skill Creator는 코딩 용어에 대한 친숙도가 다양한 사람들이 사용할 수 있습니다. 요즘에는 비개발자들도 터미널을 열기 시작했지만, 대다수 사용자는 어느 정도 컴퓨터에 익숙합니다.

상황에 맞게 소통 방식을 조절하세요. 기본적으로:

- "평가(evaluation)"와 "벤치마크(benchmark)"는 경계선이지만 사용 가능
- "JSON"이나 "어서션(assertion)"은 사용자가 알고 있다는 명확한 단서가 있을 때만 설명 없이 사용

의심스러우면 간단한 설명을 덧붙여도 됩니다.

## 스킬 만들기

### 의도 파악

사용자의 의도부터 파악하세요. 현재 대화에 이미 사용자가 캡처하려는 워크플로우가 있을 수 있습니다(예: "이걸 스킬로 만들어줘"). 그런 경우 대화 기록에서 먼저 답을 추출하세요. 사용자가 공백을 채우고 확인해야 할 수 있습니다.

1. 이 스킬로 Claude가 무엇을 할 수 있어야 하나요?
2. 언제 이 스킬이 트리거되어야 하나요? (어떤 사용자 문구/상황에서)
3. 예상 출력 형식은 무엇인가요?
4. 스킬이 제대로 작동하는지 확인하기 위한 테스트 케이스가 필요한가요? 객관적으로 검증 가능한 출력(파일 변환, 데이터 추출, 코드 생성, 정해진 워크플로우)이 있는 스킬은 테스트 케이스가 유용합니다. 주관적인 출력(글쓰기 스타일, 예술)은 대개 불필요합니다.

### 인터뷰 및 조사

엣지 케이스, 입출력 형식, 예시 파일, 성공 기준, 의존성 등에 대해 적극적으로 질문하세요. 이 부분을 정리한 후에 테스트 프롬프트를 작성하세요.

사용 가능한 MCP를 확인하고, 조사에 유용하다면 서브에이전트를 통해 병렬로 조사하거나 직접 수행하세요. 사용자의 부담을 줄이기 위해 충분한 맥락을 갖추고 접근하세요.

### SKILL.md 작성

사용자 인터뷰를 바탕으로 다음 구성 요소를 작성하세요:

- **name**: 스킬 식별자
- **description**: 언제 트리거될지, 무엇을 하는지. 이것이 주요 트리거 메커니즘입니다. "언제 사용"에 대한 정보는 모두 여기에. 참고: Claude는 스킬을 "과소 트리거"하는 경향이 있어서, 설명을 조금 더 적극적으로 작성하세요. 예를 들어 "내부 Anthropic 데이터 표시를 위한 빠른 대시보드 구축 방법" 대신 "내부 Anthropic 데이터 표시를 위한 빠른 대시보드 구축 방법. 사용자가 대시보드, 데이터 시각화, 내부 지표, 또는 회사 데이터 표시를 언급할 때 반드시 이 스킬을 사용하세요."처럼 작성하세요.
- **compatibility**: 필요한 도구, 의존성 (선택 사항, 거의 필요 없음)
- **나머지 스킬 내용**

### 스킬 작성 가이드

#### 스킬의 구조

```
skill-name/
├── SKILL.md (필수)
│   ├── YAML 프런트매터 (name, description 필수)
│   └── 마크다운 지침
└── 번들 리소스 (선택)
    ├── scripts/    - 결정론적/반복 작업을 위한 실행 코드
    ├── references/ - 필요에 따라 컨텍스트에 로드되는 문서
    └── assets/     - 출력에 사용되는 파일 (템플릿, 아이콘, 폰트 등)
```

#### 점진적 공개 (Progressive Disclosure)

스킬은 3단계 로딩 시스템을 사용합니다:
1. **메타데이터** (name + description) - 항상 컨텍스트에 (~100단어)
2. **SKILL.md 본문** - 스킬이 트리거될 때마다 컨텍스트에 (<500줄 권장)
3. **번들 리소스** - 필요에 따라 (무제한, 스크립트는 로드 없이 실행 가능)

이 단어 수는 대략적인 기준이며 필요하면 더 길어도 됩니다.

**핵심 패턴:**
- SKILL.md는 500줄 미만으로 유지. 이 한계에 근접하면 추가 계층과 명확한 다음 단계 안내를 추가하세요.
- 언제 읽어야 하는지 안내와 함께 SKILL.md에서 파일을 명확히 참조하세요.
- 대형 참조 파일(>300줄)에는 목차를 포함하세요.

**도메인 구성**: 여러 도메인/프레임워크를 지원하는 스킬은 변형별로 구성하세요:
```
cloud-deploy/
├── SKILL.md (워크플로우 + 선택)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude는 관련 참조 파일만 읽습니다.

#### 놀라움 없음의 원칙

당연한 말이지만, 스킬에는 악성 코드나 시스템 보안을 위협할 수 있는 내용이 포함되어서는 안 됩니다. 스킬의 내용은 설명했을 때 사용자를 의도적으로 놀라게 하지 않아야 합니다. 오해의 소지가 있거나 무단 접근·데이터 유출 등 악의적인 활동을 촉진하도록 설계된 스킬 생성 요청에는 응하지 마세요. "XYZ로 롤플레이"같은 건 괜찮습니다.

#### 작성 패턴

지침은 명령형으로 작성하세요.

**출력 형식 정의 예시:**
```markdown
## 보고서 구조
항상 이 정확한 템플릿을 사용하세요:
# [제목]
## 요약
## 주요 발견사항
## 권고사항
```

**예시 패턴:**
```markdown
## 커밋 메시지 형식
**예시 1:**
입력: JWT 토큰으로 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현
```

### 작성 스타일

가능하면 왜 중요한지 이유를 설명하세요. 이론적 사고를 활용하고 스킬이 특정 예시에 지나치게 좁게 맞춰지지 않도록 하세요. 초안을 먼저 작성한 후 새로운 시각으로 검토하고 개선하세요.

### 테스트 케이스

스킬 초안을 작성한 후 실제 사용자가 입력할 법한 2~3개의 현실적인 테스트 프롬프트를 만드세요. 사용자와 공유하고 확인받은 후 실행하세요.

테스트 케이스는 `evals/evals.json`에 저장하세요. 처음에는 어서션 없이 프롬프트만 작성하고, 어서션은 다음 단계에서 초안 작성합니다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자의 작업 프롬프트",
      "expected_output": "예상 결과 설명",
      "files": []
    }
  ]
}
```

전체 스키마(어서션 필드 포함)는 `references/schemas.md`를 참고하세요.

## 테스트 케이스 실행 및 평가

이 섹션은 하나의 연속 순서입니다. 중간에 멈추지 마세요. `/skill-test`나 다른 테스트 스킬을 사용하지 마세요.

결과는 스킬 디렉토리의 형제 디렉토리인 `<skill-name>-workspace/`에 저장하세요. 워크스페이스 내에서 반복별(`iteration-1/`, `iteration-2/` 등)로 결과를 구성하고, 각 테스트 케이스는 디렉토리(`eval-0/`, `eval-1/` 등)를 갖습니다.

### 1단계: 같은 턴에 모든 실행 스폰 (스킬 포함 및 기준선)

각 테스트 케이스에 대해 같은 턴에 두 개의 서브에이전트를 스폰합니다. 하나는 스킬 포함, 하나는 기준선. 이것이 중요합니다: 스킬 포함 실행을 먼저 스폰하고 나중에 기준선을 추가하지 마세요. 모든 것을 동시에 실행하세요.

**스킬 포함 실행:**
```
이 작업을 수행하세요:
- 스킬 경로: <스킬 경로>
- 작업: <eval 프롬프트>
- 입력 파일: <eval 파일이 있으면, 없으면 "없음">
- 출력 저장 위치: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 저장할 출력: <사용자가 중요하게 여기는 것 — 예: ".docx 파일", "최종 CSV">
```

**기준선 실행** (같은 프롬프트, 기준선은 상황에 따라):
- **새 스킬 만들기**: 스킬 없음. 같은 프롬프트, 스킬 경로 없음, `without_skill/outputs/`에 저장.
- **기존 스킬 개선**: 이전 버전. 수정 전에 스킬 스냅샷(`cp -r <스킬경로> <workspace>/skill-snapshot/`)을 찍고, 기준선 서브에이전트를 스냅샷에 연결. `old_skill/outputs/`에 저장.

각 테스트 케이스마다 `eval_metadata.json`을 작성하세요(어서션은 일단 비워도 됩니다). 각 eval에 테스트하는 내용을 기반으로 설명적인 이름을 붙이세요.

```json
{
  "eval_id": 0,
  "eval_name": "설명적인-이름",
  "prompt": "사용자의 작업 프롬프트",
  "assertions": []
}
```

### 2단계: 실행 중에 어서션 초안 작성

실행이 완료되기를 기다리는 동안 생산적으로 활용하세요. 각 테스트 케이스의 정량적 어서션 초안을 작성하고 사용자에게 설명하세요.

좋은 어서션은 객관적으로 검증 가능하고 설명적인 이름을 가집니다. 주관적인 스킬(글쓰기 스타일, 디자인 품질)은 정성적 평가가 더 적합합니다.

`eval_metadata.json`과 `evals/evals.json`을 어서션으로 업데이트하세요.

### 3단계: 실행 완료 시 타이밍 데이터 캡처

각 서브에이전트 작업이 완료되면 `total_tokens`와 `duration_ms`가 포함된 알림을 받습니다. 즉시 실행 디렉토리의 `timing.json`에 저장하세요:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

이 데이터는 작업 알림을 통해서만 캡처할 수 있으며 나중에 복구할 수 없습니다.

### 4단계: 채점, 집계, 뷰어 실행

모든 실행이 완료되면:

1. **각 실행 채점** — 그레이더 서브에이전트(또는 인라인)가 `agents/grader.md`를 읽고 각 어서션을 출력에 대해 평가. 결과를 각 실행 디렉토리의 `grading.json`에 저장. grading.json의 expectations 배열은 반드시 `text`, `passed`, `evidence` 필드를 사용하세요.

2. **벤치마크로 집계** — skill-creator 디렉토리에서 집계 스크립트 실행:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <이름>
   ```
   이 명령으로 `benchmark.json`과 `benchmark.md`가 생성됩니다.

3. **분석가 검토** — 벤치마크 데이터를 읽고 집계 통계가 숨길 수 있는 패턴을 발견하세요. `agents/analyzer.md` 참고.

4. **뷰어 실행**:
   ```bash
   nohup python <skill-creator-경로>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   2회차 이상이면 `--previous-workspace <workspace>/iteration-<N-1>`도 추가.

   **Cowork/헤드리스 환경:** `webbrowser.open()`을 사용할 수 없으면 `--static <출력경로>`를 사용해 서버 없이 독립 HTML 파일을 생성하세요.

5. **사용자에게 알리기**: "결과를 브라우저에 열었습니다. '출력(Outputs)' 탭에서 각 테스트 케이스를 클릭하고 피드백을 남길 수 있고, '벤치마크(Benchmark)' 탭에서 정량적 비교를 볼 수 있습니다. 완료되면 돌아와서 알려주세요."

### 뷰어에서 사용자가 보는 것

"출력(Outputs)" 탭은 한 번에 하나의 테스트 케이스를 보여줍니다:
- **프롬프트**: 실행된 작업
- **출력**: 스킬이 생성한 파일 (가능하면 인라인 렌더링)
- **이전 출력** (2회차 이상): 이전 반복의 출력 (접혀 있음)
- **공식 채점** (채점된 경우): 어서션 통과/실패 (접혀 있음)
- **피드백**: 자동 저장되는 텍스트 박스
- **이전 피드백** (2회차 이상): 지난번 댓글

탐색은 이전/다음 버튼이나 방향키로. 완료되면 "모든 리뷰 제출"을 클릭해 `feedback.json`에 저장.

### 5단계: 피드백 읽기

사용자가 완료를 알려주면 `feedback.json`을 읽으세요:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "차트에 축 레이블이 없음", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "완벽해요, 마음에 들어요", "timestamp": "..."}
  ],
  "status": "complete"
}
```

빈 피드백은 사용자가 괜찮다고 생각한다는 의미. 특정 불만이 있는 테스트 케이스에 집중해서 개선하세요.

완료 후 뷰어 서버 종료:
```bash
kill $VIEWER_PID 2>/dev/null
```

## 스킬 개선

이것이 루프의 핵심입니다. 테스트 케이스를 실행했고, 사용자가 결과를 검토했으며, 이제 피드백을 바탕으로 스킬을 개선해야 합니다.

### 개선 방법 생각하기

1. **피드백에서 일반화하기.** 여기서 일어나는 큰 그림은 수백만 번 사용될 스킬을 만들려는 것입니다. 몇 가지 예시에만 지나치게 맞춰진 스킬은 쓸모없습니다. 지나치게 세밀하거나 제한적인 변경보다는 다른 비유나 작업 패턴을 시도해 보세요.

2. **프롬프트를 간결하게 유지.** 효과 없는 내용은 제거하세요. 최종 출력뿐만 아니라 트랜스크립트를 읽으세요. 스킬이 비생산적인 작업에 많은 시간을 낭비하게 만들면, 그 부분을 제거하고 어떻게 되는지 보세요.

3. **이유를 설명하기.** 무엇을 요청하는지의 이유를 설명하세요. 오늘날의 LLM은 영리합니다. ALWAYS나 NEVER를 대문자로 쓰고 싶다면, 그건 경고 신호입니다. 가능하면 이유를 설명해서 모델이 중요성을 이해하게 하세요.

4. **테스트 케이스 간의 반복 작업 찾기.** 모든 테스트 케이스에서 서브에이전트들이 비슷한 헬퍼 스크립트를 독립적으로 작성했다면, 그 스크립트를 `scripts/`에 번들링하고 스킬에서 사용하라고 지시하세요.

이 작업은 매우 중요합니다. 서두르지 말고 충분히 생각하세요. 초안을 작성한 후 새로운 시각으로 검토하고 개선하세요.

### 반복 루프

스킬 개선 후:

1. 스킬에 개선 사항 적용
2. 모든 테스트 케이스를 새 `iteration-<N+1>/` 디렉토리에 다시 실행 (기준선 포함)
3. 이전 반복 워크스페이스를 `--previous-workspace`로 지정해 리뷰어 실행
4. 사용자가 검토하고 완료를 알려줄 때까지 대기
5. 새 피드백 읽기, 다시 개선, 반복

다음 중 하나일 때 중단:
- 사용자가 만족 표현
- 피드백이 모두 빈 경우 (모두 좋아 보임)
- 의미 있는 진전이 없는 경우

이것은 선택 사항이고 대부분의 사용자에게는 필요 없습니다. 사람의 검토 루프만으로도 충분합니다.

## 설명 최적화

SKILL.md 프런트매터의 description 필드가 Claude가 스킬을 호출할지 결정하는 주요 메커니즘입니다. 스킬을 만들거나 개선한 후 트리거 정확도를 위해 설명 최적화를 제안하세요.

### 1단계: 트리거 평가 쿼리 생성

트리거해야 하는 것과 하지 말아야 하는 것을 섞어 20개의 평가 쿼리를 만드세요. JSON으로 저장:

```json
[
  {"query": "사용자 프롬프트", "should_trigger": true},
  {"query": "다른 프롬프트", "should_trigger": false}
]
```

쿼리는 실제 사용자가 입력할 법한 현실적인 내용이어야 합니다. 파일 경로, 개인적 맥락, 칼럼 이름, 회사 이름 등 구체적인 내용을 포함하세요.

나쁜 예: `"이 데이터 형식 변환"`, `"PDF에서 텍스트 추출"`, `"차트 만들기"`

좋은 예: `"상사가 방금 xlsx 파일을 보냈는데 (다운로드에 'Q4 sales final FINAL v2.xlsx'처럼 뭔가가 있음) 수익 마진 퍼센트를 보여주는 열을 추가해달래요. C열이 수익이고 D열이 비용인 것 같아요"`

**트리거해야 하는 쿼리**(8~10개): 같은 의도의 다양한 표현 방식. 공식적인 것과 격식 없는 것 혼합. 스킬이나 파일 유형을 명시하지 않아도 명확히 필요한 케이스 포함.

**트리거하면 안 되는 쿼리**(8~10개): 가장 가치 있는 것은 아슬아슬한 케이스들입니다. 키워드를 공유하지만 실제로는 다른 것이 필요한 쿼리들.

### 2단계: 사용자와 검토

HTML 템플릿을 사용해 사용자에게 평가 세트 검토를 요청하세요:

1. `assets/eval_review.html` 템플릿 읽기
2. 플레이스홀더 교체:
   - `__EVAL_DATA_PLACEHOLDER__` → JSON 배열 (따옴표 없이 JS 변수 할당)
   - `__SKILL_NAME_PLACEHOLDER__` → 스킬 이름
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 현재 스킬 설명
3. 임시 파일에 쓰고 열기: `open /tmp/eval_review_<스킬이름>.html`
4. 사용자가 쿼리 편집, 트리거 여부 전환, 항목 추가/제거 후 "평가 세트 내보내기" 클릭
5. 파일이 `~/Downloads/eval_set.json`으로 다운로드됨

### 3단계: 최적화 루프 실행

백그라운드에서 최적화 루프를 실행하세요:

```bash
python -m scripts.run_loop \
  --eval-set <트리거-eval.json 경로> \
  --skill-path <스킬 경로> \
  --model <이 세션을 구동하는 모델 ID> \
  --max-iterations 5 \
  --verbose
```

시스템 프롬프트의 모델 ID를 사용하세요(사용자가 실제로 경험하는 것과 트리거 테스트가 일치하도록).

실행 중 주기적으로 출력을 확인해 사용자에게 진행 상황을 알려주세요.

이 도구는 전체 최적화 루프를 자동으로 처리합니다. 평가 세트를 훈련(60%)과 홀드아웃 테스트(40%)로 분할하고, 현재 설명을 평가(각 쿼리 3회 실행)한 후, 실패한 항목을 바탕으로 Claude에게 개선 제안을 요청합니다.

### 스킬 트리거 작동 방식

스킬은 Claude의 `available_skills` 목록에 name + description과 함께 나타나며, Claude는 해당 설명을 바탕으로 스킬을 참고할지 결정합니다. Claude는 혼자서 쉽게 처리할 수 없는 작업에 대해서만 스킬을 참고합니다. 따라서 평가 쿼리는 Claude가 실제로 스킬을 참고함으로써 이득을 얻을 만큼 충분히 복잡해야 합니다.

### 4단계: 결과 적용

JSON 출력의 `best_description`을 가져다가 스킬의 SKILL.md 프런트매터를 업데이트하세요. 사용자에게 전후를 보여주고 점수를 보고하세요.

### 패키징 및 제공 (`present_files` 도구가 있는 경우만)

`present_files` 도구에 접근 가능한지 확인하세요. 없으면 이 단계를 건너뛰세요. 있으면 스킬을 패키징하고 .skill 파일을 사용자에게 제공하세요:

```bash

```

## Claude.ai 전용 지침

Claude.ai에서는 핵심 워크플로우는 동일하지만 서브에이전트가 없어 일부 메커니즘이 달라집니다:

**테스트 케이스 실행**: 서브에이전트 없음. 각 테스트 케이스에 대해 SKILL.md를 읽은 후 지침을 따라 직접 수행. 직렬로 하나씩. 기준선 실행 건너뜀.

**결과 검토**: 브라우저를 열 수 없으면 뷰어 건너뛰고 대화에서 직접 결과 제공. 파일인 경우 파일시스템에 저장하고 위치를 알려주어 사용자가 다운로드·확인하게. 인라인으로 피드백 요청.

**벤치마킹**: 서브에이전트 없이는 정량적 벤치마킹 건너뜀. 정성적 피드백에 집중.

**설명 최적화**: `claude -p`가 필요하며 Claude Code에서만 사용 가능. Claude.ai에서는 건너뜀.

**블라인드 비교**: 서브에이전트 필요. 건너뜀.

**기존 스킬 업데이트**:
- 원래 이름 유지. 스킬의 디렉토리 이름과 `name` 프런트매터 필드를 확인하고 그대로 유지.
- 수정 전 쓰기 가능한 위치에 복사. `/tmp/skill-name/`에 복사하고 거기서 수정.

Cowork에서 알아야 할 주요 사항:

- 피드백은 다르게 작동합니다: 서버가 없으므로 뷰어의 "모든 리뷰 제출" 버튼이 `feedback.json`을 파일로 다운로드합니다.
- 패키징 작동합니다.
- 설명 최적화(`run_loop.py`)는 `claude -p`를 서브프로세스로 사용하므로 Cowork에서도 작동합니다.
- **기존 스킬 업데이트**: Claude.ai 섹션의 업데이트 가이드를 따르세요.

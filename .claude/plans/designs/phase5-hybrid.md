# Phase 5 사전설계 — 하이브리드 구조 전환

> 작성일: 2026-04-06
> 상태: 사전설계 (Phase 4 완료 후 활성화)
> 목적: AI 호출 최소화로 운영 비용 80~85% 절감
> 전제: Phase 4 완료 + 실사용 데이터 확보 후 진입

---

## 전환 배경

- Phase 4 현재 구조: 모든 턴 → AI 호출 (10회/상담)
- 설계 철학은 이미 "약속된 질문 출력" 중심 — AI 의존도 낮음
- 하이브리드 전환 = 프롬프트가 AI에게 시키던 구조 제어를 코드로 명시화
- 실사용량 확인 후 비용 절감 필요성 판단 시 진입

---

| 단계 | 처리 방식 |
|------|---------|
| 시작/웰컴 | 오픈빌더 웰컴 블록 고정 텍스트 |
| 거래당사자 선택 | 오픈빌더 버튼 4개 |

| 확인 화면 | 서버가 수집값 포맷팅 후 출력 |
| 이전/다시/취소 | 서버 코드 직접 처리 |

| 단계 | 이유 |

| 주소/희망조건 입력 | 자연어 파싱 필요 |
| 특이사항/어필포인트 | 자유 서술 |
| 고객정보 (성함/연락처) | 형식 검증 + 자연어 |
| 잘못된 입력 처리 | 예외 안내 |

### 코드 변경

- `app/services/flow.py` 신규 — 상태기계 (FLOW_STATES, PROPERTY_QUESTIONS)
- `app/routes/kakao.py` 수정 — 상태에 따라 AI 호출 분기
- `app/services/ai.py` 수정 — 자유입력 처리 전용 축소 프롬프트
- `prompts/완성프롬프트.md` 대폭 축소 (~2,500토큰 → ~400토큰)

### 카카오 오픈빌더 변경

- 거래당사자 선택 블록 신규 (버튼 4개)
- 매물종류 선택 블록 신규 (버튼 10개)
- 스킬 서버 연결 구조 재설계

### 데이터 구조 변경

- `extract_request_summary()` 제거 — 상태기계가 이미 수집한 값을 직접 시트에 기록
- `data/property_features/*.md` → 서버 코드 질문 리스트로 전환 또는 병행 유지

## 설계 시 반드시 해결해야 할 문제 (2026-04-06 협의)

> **새 세션 AI를 위한 경고**: 아래 문제들은 단순 구현 이슈가 아니라 구조 설계 차원의 문제다.
> "약속된 질문지 출력" 방식으로 전환하면 현재 AI 자율 구조에서 없던 새로운 문제가 발생한다.
> 각 문제를 해결하지 않고 구현하면 실사용 불가 수준의 오동작이 발생한다.

### 문제 1 — 돌발 답변 처리

**문제 취지:**
약속된 질문지를 코드/파일에서 고정 출력하면, 사용자가 예상 범위 밖의 답변을 입력했을 때 처리 방법이 없다.
현재 AI 자율 구조에서는 AI가 알아서 처리하지만, 하이브리드 구조에서는 코드가 흐름을 제어하므로 예외 처리를 명시적으로 설계해야 한다.

**해결해야 할 내용:**
- 약속된 답변 범위(번호 선택, 특정 키워드 등) 외 입력 감지 기준 정의
- 돌발 입력 발생 시 AI 연계 방식 설계 (AI에게 넘기는 범위와 방식)
- AI 연계 후 다시 약속된 흐름으로 복귀하는 방법
- 무한 돌발 루프 방지 (사용자가 계속 예상 밖 입력을 할 경우)

### 문제 2 — 초기화 현상

**문제 취지:**
현재 AI 자율 구조에서도 답변 진행 중 처음으로 돌아가는 버그가 발생한다.
하이브리드 구조에서 상태기계로 전환하면 상태 관리가 명시화되므로 이 문제를 원천 해결할 수 있어야 한다.
해결하지 않으면 사용자 신뢰가 무너진다.

**해결해야 할 내용:**
- 상태기계 상태값 저장 방식 (세션 내 상태 유실 방지)
- "이전" 키워드 처리 시 상태 스택 관리
- 서버 재시작 시 세션 초기화는 허용하되, 정상 흐름 중 초기화는 방지

### 문제 3 — 매물유형 선택: 번호 vs 직접 입력

**문제 취지:**
사용자가 번호(예: 1)로도 입력하고 명칭(예: "아파트", "원룸")으로도 입력한다.
약속된 질문지 구조에서는 입력값을 코드가 파싱해야 하므로, 번호와 자연어 명칭을 모두 처리하는 매핑 로직이 필요하다.

**해결해야 할 내용:**
- 번호 입력: 1→주거용(분양형), 2→주거용(단독건물) 등 직접 매핑
- 자연어 입력: "아파트"→1번, "빌라"→1번, "단독"→2번 등 매핑 테이블
- 매핑 실패(인식 불가) → 돌발 답변 처리 연계
- 복수 선택 처리: "1, 3" 또는 "아파트, 원룸" 형식

### 문제 4 — 서브타입 분기 처리

**문제 취지:**
매물유형 1번(주거용 분양형) 안에 아파트/주상복합/오피스텔/도시형/다세대가 있다.
각각 질문 항목이 미묘하게 다르다 (단지명 vs 건물명, 동호수 유무 등).
약속된 질문지 구조에서 이 분기를 어떻게 처리할지 설계가 없으면 단일 항목 목록밖에 못 만든다.

**해결해야 할 내용:**
- 1번 선택 후 서브타입 추가 질문 여부 결정 (아파트인지 오피스텔인지 확인 필요)
- 서브타입별 질문 항목 분기 로직
- 서브타입 자동 감지 가능 여부 (단지명에서 유추) vs 명시적 선택 요구

### 문제 5 — 약속된 흐름 vs AI 연계의 경계 정의

**문제 취지:**
어디까지 코드/파일로 처리하고 어디서 AI를 호출할지 경계가 명확하지 않으면
구현할 때마다 판단이 달라지고 일관성이 깨진다.
현재 phase5-hybrid.md의 턴 분류는 개략적인 수준이며, 매물유형별로 세부 경계가 달라질 수 있다.

**해결해야 할 내용:**
- 매물유형별 AI 호출 시점 명확화
- 자유입력 턴(주소, 특이사항 등)에서 AI가 받는 컨텍스트 최소화 방법
- 구글 시트 기록: 상태기계가 직접 수집한 값을 바로 기록 (extract_request_summary 제거)
  - 취지: 현재 extract_request_summary가 대화 이력에서 AI로 추출하는 방식은 다수 세션에서 접수ID+일시만 기록되는 문제가 반복됨. 방향 자체가 잘못됨. 하이브리드 전환 시 이 함수를 제거하고 상태기계가 직접 수집한 값을 기록하는 방식으로 전환하면 원천 해결 가능.

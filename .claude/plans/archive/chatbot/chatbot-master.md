## 구현 방향 원칙 (2026-04-06 확정)

> **새 세션 AI를 위한 원칙** — 협의 결과만이 아니라 왜 이 원칙을 세웠는지 이유도 함께 기록한다.

**5단계 진행 원칙:**
1. **완성** — 문제가 있어도 일단 배포 가능 수준까지 완성
2. **이슈 해결** — 완성된 결과로 드러난 이슈 해결
3. **테스트/피드백** — 실사용자 테스트 및 피드백 수집
4. **결과 보완** — 피드백 기반 반복 보완
5. **배포** — 최종 배포

**왜 이 원칙인가:**
- Phase 3는 "완성"이 아니라 "테스트 없이 결정할 수 없는 부분을 미룬 상태"였다. 실사용 테스트 없이 질문 내용을 확정하는 것은 불가능하다.
- 완성 전에 이슈를 해결하려 하면 기준점이 없어 방향을 잃는다. 완성 후 이슈가 드러나야 올바른 방향으로 해결할 수 있다.

**플랜 기록 원칙:**
- 결정사항과 함께 반드시 **왜 그 결정을 했는지 취지**를 기록한다.
- 취지 없이 결정사항만 기록하면 새 세션 AI가 맥락 없이 잘못된 방향으로 해석한다.

## Phase별 완료 조건 (요약)

### Phase 1 ✅ 완료
- git 초기화 및 GitHub 레포 생성
- 프롬프트 파일 구조 설계 확정

- Claude Desktop 테스트 스킵 → FastAPI 전환 결정
→ 상세: `phase1-setup.md`

### Phase 2
- 최소 FastAPI 로컬 서버 구축
- 매물특징 파일 분기 구조 구현

- 키워드(다시·이전) 동작 확인
→ 상세: `phase2-flow.md`

### Phase 3
- 파일 구조 정리 (prompts/, data/property_features/)
- 완성프롬프트.md 보완 (매물특징 지침·고객정보·8·9번 제외·접수완료 메시지)
- 매물특징 파일 내용 보완 (사용자 제공)
- 로컬 동작 확인
→ 상세: `phase3-complete.md`

### Phase 4
- FastAPI 서버 + Google Sheets + Gemini API + 카카오 연동
- Railway 배포
- 실사용자 테스트 통과
→ 사전설계: `designs/phase4-dev.md` (Phase 4 활성화 시 plans/에 신규 작성)

- Phase 4 완료 + 실사용 데이터 확보 후 진입
→ 사전설계: `designs/phase5-hybrid.md`
